# Portable secrets — OpenBao + External Secrets Operator (DARK / staged)

ADR-0022 (Cloud-Agnostic Multi-Substrate Portability), **roadmap item 1: Portable
secrets**. Lifted from the sibling **oyatie** project (full-license reuse:
`infra/kms/openbao.k8s.yaml` + `infra/external-secrets/*`, ADR-0043 topology).

This directory replaces the out-of-band `kubectl create secret` flow
(`deploy/SECRETS.md`) with GitOps-reconciled secrets: OpenBao is the barrier-
encrypted source of truth, and the External Secrets Operator (ESO) projects each
existing k8s Secret from OpenBao KV — **into the same Secret names + keys the pods
already consume, so running workloads need zero change.**

## This is ADDITIVE and DARK

- **Nothing here auto-deploys.** The root app-of-apps (`deploy/argocd/root.yaml`)
  only recurses `deploy/argocd/apps`. These Application manifests live under
  `deploy/apps/secrets-management/`, so ArgoCD never adopts them. **Merging this
  PR changes nothing in the cluster.**
- **OCI Vault stays the source of truth** and the hand-created Secrets keep the
  app running until — and after — the founder-gated cutover below. The cutover is
  **reversible**: delete the ExternalSecrets and the pre-existing Secrets remain.
- OCI remains a first-class supported substrate. This just makes secrets portable
  to the on-prem substrate too.

## What gets projected (Secret name -> keys -> OpenBao KV path)

| k8s Secret | Keys | Consumed by | OpenBao KV path (`secret` mount, KV v2) |
|---|---|---|---|
| `mnt-secrets` | `MNT_JWT_PRIVATE_KEY_PEM`, `MNT_JWT_PUBLIC_KEY_PEM`, `MNT_S3_ACCESS_KEY_ID`, `MNT_S3_SECRET_ACCESS_KEY`, `MNT_MAIL_MASTER_KEY` (+ optional `MNT_EMAIL_SMTP_*`, `MNT_COLDSTART_OTP`, `MNT_FCM_*`, `MNT_SOLAPI_*`) | `mnt-app`, `mnt-worker` (`envFrom`) | `maintenance/app` (whole entry, `dataFrom.extract`) |
| `mnt-db-rt` | `username`, `password`, `uri` | `mnt-app`, `mnt-worker` (`DATABASE_URL`), CNPG managed `mnt_rt` role | `maintenance/db/rt` |
| `oci-objectstore-creds` | `ACCESS_KEY_ID`, `ACCESS_SECRET_KEY` | CNPG Barman `ObjectStore` (DB backups) | `maintenance/objectstore` (fields `access_key_id`, `access_secret_key`) |

**Not projected:** `mnt-db-app` — the owner/migration Secret is auto-generated by
CloudNativePG, not created out-of-band, so ESO leaves it alone.

## The auth-delegator / `disable_local_ca_jwt` invariant (read this first)

The whole ESO->OpenBao binding hinges on **one** OpenBao config flag:

```
auth/kubernetes/config  disable_local_ca_jwt=true
```

- With `disable_local_ca_jwt=true`, OpenBao performs the Kubernetes **TokenReview
  using the CLIENT JWT** — the `external-secrets/external-secrets` ServiceAccount
  token ESO presents on each login.
- That client SA is granted `system:auth-delegator` by
  `wiring/clusterrolebinding-external-secrets-auth-delegator.yaml`, so the
  TokenReview is authorized.
- This **avoids** granting `system:auth-delegator` to OpenBao's OWN SA (it runs as
  a plain Deployment SA in `mnt-kms`, not wired for delegated TokenReview).
  Leaving the flag at its default (`false`) makes OpenBao TokenReview with ITS
  local SA token, which **403s and breaks every ExternalSecret** bound to this
  store. If you ever flip it back to `false`, you MUST instead grant
  `system:auth-delegator` to OpenBao's SA in `mnt-kms`.

## Activation runbook (founder-gated ops — do NOT run on merge)

Chart versions pinned: **external-secrets `2.7.0`** (app v2.7.0), **OpenBao image
`ghcr.io/openbao/openbao:2.5.5`** (Helm chart appVersion parity). `bao` and
`vault` CLIs are interchangeable against OpenBao.

### 0. Prereq — let ArgoCD serve the staged Applications

Add the ESO chart repo to the AppProject allow-list (one-time; deploys nothing on
its own):

```sh
# deploy/argocd/project.yaml -> spec.sourceRepos, add:
#   - https://charts.external-secrets.io
```

Then activate the three staged Applications, either by moving them into the
auto-synced dir:

```sh
git mv deploy/apps/secrets-management/application-*.yaml deploy/argocd/apps/
```

…or by applying them directly (`kubectl apply -f deploy/apps/secrets-management/application-external-secrets.yaml` etc.).
ESO (wave -1) + OpenBao (wave -1) come up; `secrets-wiring` (wave 1) applies the
ClusterSecretStore/CRB/ExternalSecrets, which stay **NotReady** until the OpenBao
steps below — expected and harmless (OCI Vault + the existing Secrets keep serving).

### 1. Initialize + unseal OpenBao

```sh
kubectl -n mnt-kms exec -it deploy/openbao -- bao operator init -key-shares=5 -key-threshold=3
```

**Store the 5 unseal keys + the root token OUT of the cluster** — OCI Vault or an
offline tamper-evident store. **NEVER** `/tmp`, git, tickets, or chat. Then unseal
(repeat with 3 distinct keys):

```sh
kubectl -n mnt-kms exec -it deploy/openbao -- bao operator unseal   # x3
```

### 2. Enable the KV v2 mount + Kubernetes auth

```sh
export BAO_TOKEN=<root-token>   # from step 1; export only in this shell, never persist
kubectl -n mnt-kms exec -it deploy/openbao -- sh -c '
  bao secrets enable -version=2 -path=secret kv
  bao auth enable kubernetes
'
```

### 3. Configure the Kubernetes auth method (**the invariant**)

```sh
kubectl -n mnt-kms exec -it deploy/openbao -- bao write auth/kubernetes/config \
    kubernetes_host="https://kubernetes.default.svc" \
    kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt \
    disable_local_ca_jwt=true \
    disable_iss_validation=true
```

- `disable_local_ca_jwt=true` — see the invariant above (CLIENT JWT does the
  TokenReview; the ESO SA holds `system:auth-delegator`).
- `disable_iss_validation=true` — tolerates projected-token / bound-SA-token
  issuer differences across cluster setups.

### 4. Read policy + role bound to the ESO ServiceAccount

```sh
kubectl -n mnt-kms exec -it deploy/openbao -- sh -c '
  bao policy write mnt-eso-read - <<EOF
path "secret/data/maintenance/*" { capabilities = ["read"] }
EOF
  bao write auth/kubernetes/role/eso-mnt \
      bound_service_account_names=external-secrets \
      bound_service_account_namespaces=external-secrets \
      policies=mnt-eso-read \
      ttl=1h
'
```

### 5. Seed the KV paths from the CURRENT OCI-Vault secret values

Copy the values that are live today (read them from OCI Vault / the existing k8s
Secrets — `kubectl -n maintenance get secret mnt-secrets -o jsonpath=...`). Store
**field names exactly as the k8s Secret keys** for the `dataFrom.extract` path.

```sh
# mnt-secrets (fields MUST be the exact env-var names; add optional ones as they go live)
bao kv put secret/maintenance/app \
    MNT_JWT_PRIVATE_KEY_PEM=@jwt-private.pem \
    MNT_JWT_PUBLIC_KEY_PEM=@jwt-public.pem \
    MNT_S3_ACCESS_KEY_ID="<oci-access-key>" \
    MNT_S3_SECRET_ACCESS_KEY="<oci-secret-key>" \
    MNT_MAIL_MASTER_KEY="<base64-32-byte-key>"

# mnt-db-rt (uri = postgresql://mnt_rt:<pw>@mnt-db-rw.maintenance.svc:5432/maintenance)
bao kv put secret/maintenance/db/rt \
    username="mnt_rt" password="<rt-password>" uri="<rt-uri>"

# oci-objectstore-creds (S3-compatible OCI Customer Secret Key; lowercase fields)
bao kv put secret/maintenance/objectstore \
    access_key_id="<oci-access-key>" access_secret_key="<oci-secret-key>"
```

> Never commit these values. They live only in OpenBao (barrier-encrypted at rest)
> and are projected by ESO. `rm -f jwt-*.pem` after seeding.

### 6. Verify the projected Secrets match what pods expect

```sh
# store Ready
kubectl get clustersecretstore openbao-maintenance -o jsonpath='{.status.conditions}'

# each ExternalSecret -> SecretSynced, and the target Secret exists with the right keys
for es in mnt-secrets mnt-db-rt oci-objectstore-creds; do
  kubectl -n maintenance get externalsecret "$es"
  kubectl -n maintenance get secret "$es" -o jsonpath='{.data}' | tr ',' '\n'
done
```

Confirm the key set equals what `deploy/SECRETS.md` documents (esp. that
`mnt-secrets` has all 5 required keys and `mnt-db-rt` has `username/password/uri`).
A `403`/`permission denied` on the store almost always means the
`disable_local_ca_jwt` / auth-delegator invariant is broken (flag `false`, or the
ClusterRoleBinding missing).

### 7. Retire the out-of-band step (only after step 6 is green)

Once ESO owns the Secrets and pods are healthy, stop hand-creating them: remove
the `kubectl create secret` instructions from your ops flow and let ESO reconcile.
Rotating a value now = `bao kv put` the KV path; ESO repropagates within
`refreshInterval` (1h) with **no workload redeploy**.

**Rollback (reversible):** delete the `secrets-wiring` Application (or the
individual ExternalSecrets). `creationPolicy: Owner` means the Secrets it created
go with it — so re-create them out-of-band per `deploy/SECRETS.md` first if you
roll back after retiring the manual step. OCI Vault is untouched throughout.

## HA / scaling note

OpenBao runs **single-node with `storage "file"`** here (one small OCI node). The
ADR-0022 bare-metal `on-prem` target runs it as a **3-node Raft HA cluster** —
switch `storage "file"` → `storage "raft"`, `replicas: 1` → `3`, and add a
headless Service for peer discovery. The ESO wiring is unchanged by that switch.
