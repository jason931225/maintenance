**Objective:** Avoid a manifest/lock mismatch if npm expects `web` workspace version consistency.

**Check:**



**Expected:** Either `package-lock.json` has no `web` package version entry, or it matches `web/package.json` after the lock-only update.

**Approval gate:** Include `package-lock.json` in the release PR only if it changes solely due to version metadata; do not accept dependency resolution churn.

## Risks and mitigations

- **Release Please token behavior:** Comments note that tags pushed with `GITHUB_TOKEN` may not trigger downstream tag workflows. Verify `RELEASE_PLEASE_TOKEN` is configured or be prepared for approved manual dispatch.
- **Version scope ambiguity:** Only `web/package.json` is tied to the repo release. `clients/ts` and `coss-rn` stay `0.1.0` unless release policy changes.
- **Lockfile drift:** `web/package.json` version bump may require `package-lock.json` update. Use lock-only update and reject dependency churn.
- **Deployment coupling:** Production digest changes are automated after image build/scan/sign. Do not combine manual digest pins with the version bump unless this is an emergency release.
- **CI path filters:** Release-only metadata changes may not trigger image builds unless tag behavior triggers as expected. Confirm Actions runs after merge.
- **Pre-1.0 semver:** Config has `bump-minor-pre-major` and `bump-patch-for-minor-pre-major`; verify generated version matches expectations for feat/fix commits before approval.
