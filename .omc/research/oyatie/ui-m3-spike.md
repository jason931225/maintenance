# UI-M3 Overview Blueprint

Artifact path: `.omc/research/oyatie/ui-m3-spike.md`

## Verdict

**Full zero-backend-change UI-M3 is INFEASIBLE.** The overview can replace `/work-hub` inside the existing `ConsoleShell` and can consume the merged Engine-Gen workflow instance REST (`backend/openapi/openapi.yaml:6444-6685`) plus notification center list/read REST (`backend/crates/notifications/rest/src/lib.rs:24-31`, `backend/crates/notifications/rest/src/lib.rs:76-112`) without new backend work. The full charter also requires todos CRUD and a unified dispatch-offer/attendance-exception feed; the plan names todos as a new domain, and the current dispatch UI explicitly says there is no list endpoint for pending dispatch offers (`.omc/plans/oyatie-console-plan.md:77-79`, `web/src/features/dispatch/MechanicDispatchOffers.tsx:25-30`, `web/src/pages/DispatchPage.tsx:114-116`). Therefore the shippable path is: land the shell-safe overview replacement with existing real feeds, list backend gaps honestly, and add backend increments only where the charter has no source endpoint.

## Source Authority

- Design authority: the prototype overview is the `scrOverview` screen with header, KPI strip, action inbox, kind filters, grouped rows, primary actions, done state, empty/restore affordance, and keyboard hints (`docs/design/oyatie-console/Oyatie Console.dc.html:318-423`). The global design grammar requires lists to support J/K/Enter, shared tracks, drag, bottom fade/overscroll discipline, and icon-collapsing buttons (`docs/design/oyatie-console/DESIGN.md:80-82`).
- Window grammar: cards and panels use default, pinned split panel, popout float, and minimized tray states; pinning must reflow the body instead of overlaying it (`docs/design/oyatie-console/DESIGN.md:82-87`). Object clicks open a pinned detail panel and objects must carry code/link/audit semantics (`docs/design/oyatie-console/DESIGN.md:89-90`, `docs/design/oyatie-console/DESIGN.md:123-126`).
- Plan charter: two-shell coexistence is the chosen strangler path, with `ConsoleShell` hosting migrated mounted-persistent screens while `AppShell` keeps legacy routes (`.omc/plans/oyatie-console-plan.md:30-40`). UI-M1b already introduced `/work-hub` and `/attendance` in `ConsoleShell`, and UI-M3 later replaces `/work-hub` with the overview (`.omc/plans/oyatie-console-plan.md:65-67`, `.omc/plans/oyatie-console-plan.md:77-79`).
- Program rules: use `console-*` tokens, keep Zustand scoped to workspace state, keep data fetching on the existing OpenAPI client, and phrase workflow authorization as legacy-enforce with Cedar shadow-only (`.omc/plans/oyatie-console-plan.md:36-39`, `backend/openapi/openapi.yaml:6521-6528`, `backend/openapi/openapi.yaml:6592-6598`).

## ConsoleShell Integration

UI-M3 should be a mounted-persistent `ConsoleShell` screen, not an `AppShell` route. `AppRouter` already preserves the `ConsoleShell` instance across `/work-hub` and `/attendance` with pathless children, and says every other route remains on `AppShell` during two-shell coexistence (`web/src/AppRouter.tsx:306-321`). `ConsoleShell` currently maps `/work-hub` to the `work-hub` screen, mounts `/work-hub` and `/attendance` at once, and toggles inactive screens with visibility rather than unmounting (`web/src/components/shell/ConsoleShell.tsx:36-47`, `web/src/components/shell/ConsoleShell.tsx:237-253`, `web/src/components/shell/ConsoleShell.tsx:320-322`). UI-M3 should replace the `/work-hub` slot contents with `OverviewPage` while preserving this mounted-slot behavior.

Pinning from overview rows should follow the current context path. `ConsoleShell` provides `ConsoleScreenContext` and owner context to mounted screens (`web/src/components/shell/ConsoleShell.tsx:4-14`), and `PinButton` reads that screen context, checks owner consistency, then calls `pin(screen, object)` (`web/src/components/shell/workspace/PinButton.tsx:12-33`). The workspace store keeps Zustand limited to panel/window state and delegates mutations to pure reducers (`web/src/features/workspace/store.ts:1-4`, `web/src/features/workspace/store.ts:80-99`), satisfying AD-2. Overview rows therefore map real API rows into `PinnedObject` values and render the same `PinButton`; primary actions remain separate real mutations.

Quadrant behavior stays owned by the window engine. The workspace type model is a 2x2 quadrant grid where the body section takes the largest free rectangle, pinned panels occupy quadrant/half areas, floats are fixed popouts, and minimized panels live in the tray (`web/src/features/workspace/types.ts:1-7`, `web/src/features/workspace/types.ts:14-28`, `web/src/features/workspace/types.ts:59-80`). `QuadrantContainer` computes the free section area from pinned panels and keeps the page body mounted even when panels fill the grid (`web/src/components/shell/workspace/QuadrantContainer.tsx:9-13`, `web/src/components/shell/workspace/QuadrantContainer.tsx:29-50`). `pinObject` dedupes same screen/object pins, evicts overlapping same-screen pinned panels to minimized state, and leaves other screens untouched (`web/src/features/workspace/reducer.ts:31-72`); tests lock those rules (`web/src/features/workspace/reducer.test.ts:22-68`). UI-M3 must not implement its own split layout.

Workspace persistence is server-owned and per person. The hook stores a schema-versioned `layout` under `GET/PUT /api/v1/me/workspace`, sanitizes untrusted blobs on load, and intentionally avoids `localStorage` (`web/src/features/workspace/persistence.ts:1-10`, `web/src/features/workspace/persistence.ts:26-41`). It loads once, hydrates sanitized panels, debounce-saves panel changes, and disables saves after a failed load so a transient read failure cannot clobber the server layout (`web/src/features/workspace/persistence.ts:183-234`). Tests cover save-after-load, no-save-after-failed-load, merge-with-edits, and owner reset (`web/src/features/workspace/persistence.test.ts:108-166`, `web/src/features/workspace/persistence.test.ts:235-255`).

Persisted `/work-hub` pins need an explicit migration decision. `SCREEN_KEYS` currently allow only `work-hub` and `attendance` (`web/src/features/workspace/types.ts:8-9`), and the sanitizer drops panels whose `screen` is not in that allowlist (`web/src/features/workspace/sanitize.ts:93-107`). If UI-M3 introduces `overview` as a new screen key, existing `work-hub:*` pins would stay under the old key unless migrated; if `work-hub` is reused as the screen key while the route label becomes "통합 개요", existing pins survive automatically. The smallest shippable path is to keep the persisted screen key `work-hub` for the replacement increment, change the nav/i18n label to "통합 개요", and add a later schema-v2 migration only if product requires a semantic `overview` key.

## Routing And Navigation Decision

**Chosen: direct route-slot replacement plus nav remap.** Keep `/work-hub` as the public route and keep `screen: "work-hub"` as the workspace persistence key, but replace the mounted `WorkHubPage` body with `OverviewPage` and change the visible nav/i18n label from "업무 허브" to "통합 개요". This matches the current router: `/work-hub` and `/attendance` are pathless children under one preserved `ConsoleShell`, while all other routes stay on `AppShell` during two-shell coexistence (`web/src/AppRouter.tsx:306-321`). It also matches the current shell slot: `SCREEN_FOR_PATH` maps `/work-hub` to `work-hub`, `NAV_ITEM_FOR_SCREEN` maps that screen to the `work-hub` nav item, and the mounted slot currently renders `WorkHubPage` (`web/src/components/shell/ConsoleShell.tsx:36-47`, `web/src/components/shell/ConsoleShell.tsx:86-103`, `web/src/components/shell/ConsoleShell.tsx:237-245`). The nav item already owns `key: "work-hub"`, `href: "/work-hub"`, and `labelKey: "nav.work-hub"` (`web/src/components/shell/nav.ts:344-354`), and labels resolve through `ko.ts` (`web/src/components/shell/nav-labels.ts:1-12`, `web/src/i18n/ko.ts:78-80`).

**Rejected: redirect `/work-hub` to `/overview`.** A redirect creates a new route and a new route identity without solving any UI-M3 requirement. The authenticated landing paths currently fall back to `/work-hub`: login defaults there, onboarding navigates there after passkey enrollment, integrity denial redirects there, and the authenticated `AppShell` catch-all redirects there (`web/src/pages/LoginPage.tsx:21-24`, `web/src/pages/OnboardingPage.tsx:121-147`, `web/src/components/RequireIntegrityRoute.tsx:28-36`, `web/src/AppRouter.tsx:430-431`). Redirecting would require touching all those entry points plus route smoke tests, while still needing a decision for persisted `work-hub:*` panel ids. It is higher migration risk with no extra design fidelity.

**Rejected for UI-M3 boundary: parallel `/overview` route coexistence.** A parallel route would add a third mounted-persistent `ConsoleShell` screen alongside `work-hub` and `attendance`. That expands `SCREEN_KEYS`, nav gating, sanitizer behavior, layout persistence, and tests before the replacement is proven (`web/src/features/workspace/types.ts:8-9`, `web/src/features/workspace/sanitize.ts:93-120`, `web/src/features/workspace/sanitize.test.ts:42-67`). Because `ScreenSlot` keeps mounted screen state alive for active shell children (`web/src/components/shell/ConsoleShell.tsx:320-349`), a parallel `overview` screen would also leave legacy WorkHub chrome reachable until a later cutover, violating the UI-M3 "replaces /work-hub" charter (`.omc/plans/oyatie-console-plan.md:77-79`).

The only semantic cost of the chosen route is that storage and code still say `work-hub` while the user sees "통합 개요". That is acceptable for the replacement increment because the persisted object ids already use the screen key (`web/src/features/workspace/reducer.test.ts:22-31`), and a semantic rename can be a later schema-v2 migration if product wants `/overview` and `overview:*` ids.

## Work-Hub Replacement

1. Keep `/work-hub` as the initial route target. Login, onboarding, integrity redirects, and catch-all routes already point at `/work-hub` (`web/src/pages/LoginPage.tsx:23`, `web/src/pages/OnboardingPage.tsx:125-147`, `web/src/components/RequireIntegrityRoute.tsx:35`, `web/src/AppRouter.tsx:430`).
2. Replace `WorkHubPage` with `OverviewPage` inside the existing `ScreenSlot` instead of adding parallel chrome (`web/src/components/shell/ConsoleShell.tsx:237-245`).
3. Rename the visible nav/i18n from "업무 허브" to "통합 개요" through `web/src/i18n/ko.ts`, because Korean UI strings already live there (`web/src/i18n/ko.ts:78-80`). Avoid hard-coded Korean in components.
4. Preserve existing persisted pins by keeping `screen: "work-hub"` through the replacement. Do not create a new screen key until a tested persistence migration converts existing `work-hub:*` panel ids to `overview:*`.
5. Maintain two-shell coexistence: only `/work-hub`'s mounted screen body changes; unrelated `AppShell` routes remain untouched (`.omc/plans/oyatie-console-plan.md:30-40`, `web/src/AppRouter.tsx:306-321`).
