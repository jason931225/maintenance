## 7. Generic lifecycle engine (`lcSeed`/`lcOf`/`lcSet` + single modal) вҖ” POST-SNAPSHOT

Per AGENTS В§3 and changelog 2026-07-04(11)(12)(13) [org-change reference implementation] вҶ’ 2026-07-08(10) [generalization] вҶ’ 2026-07-08(16)(18) [pendingRev staging + audit sweep] вҶ’ 2026-07-08(11) [merged into object-card layer].

**Lifecycle stages** (the "DraftвҶ’Archive" standard, DESIGN В§3.9, applies to MOST business objects): мҙҲм•Ҳ(draft) вҶ’ мғҒмӢ В·кІҖнҶ (submit/review) вҶ’ мҠ№мқёВ·кІҢмӢң(approve/publish, sensitive=passkey) вҶ’ нҷңм„ұ(active) вҶ’ к°ңм •[лІ„м „](revise/version) вҶ’ ліҙкҙҖ(archive) вҶ’ нҸҗкё°(dispose). Every transition: audited, versioned+rollback-capable, PBAC-gated, rendered as a status chip/stepper.

**Governance refinements** (DESIGN В§3.9.1-3, changelog 2026-07-04(12)): future effective-dating, pre-transition impact-analysis ("мҳҒн–Ҙл¶„м„қ мӮ¬м „м җкІҖ" вҖ” who/what is affected before you commit), maker-checker/SoD (SOX-style вҖ” the approver cannot be the initiator), DoA (м „кІ°к·ңм •, delegation-of-authority limits), referential-integrity settlement gates (dependent objects must be settled/reassigned before a parent can be disposed), change-freeze windows, and critically: **archive = hidden, never hard-delete** ("ліҙкҙҖ=мҲЁк№Җ(н•ҳл“ңмӮӯм ң кёҲм§Җ)").

**Generalized engine methods** (named in AGENTS В§3 + changelog (10), NOT read from source вҖ” spec only):
- `lcSeed(code)` вҖ” derives a lifecycle record from existing registries (`egressDocs`/`DOCS`/`docRegistered`/`ingestJobs`) rather than a separate storage table; only mutations persist via `lcReg`.
- `lcOf(code)` / `lcSet(code, patch)` вҖ” read/write the current lifecycle state for an object code.
- `lcSyncRegistries()` вҖ” bidirectional sync back into `egressDocs` state so the mail egress-gate (item 10 below) sees consistent lifecycle status.
- Transition methods: `lcSubmit`/`lcApprove`/`lcEffectuate`/`lcRevise`/`lcRollback`/`lcArchive`/`lcDep`(?)/`lcGate`/`lcDispose` вҖ” every one calls `logEvent`.
- **Single shared modal** (reusing the org-change modal's structure): 5-step stepper, effective-date field (editable only in draft), version history + non-destructive rollback, SoD approval chain, archive gate (3-item referential-integrity checklist), dispose gate (retention period + legal hold + records-manager sign-off), linked-object chips (еҺҹжң¬ provenance / ingest DX- / graph node).
- **Revision = sandbox** (changelog (16)/(10)): editing an ACTIVE object creates a v+1 DRAFT; the object shows "к°ңм • лҢҖкё° v+1 В· нҳ„н–ү мң м§Җ" (pending-revision, current version still live) until an explicit "м Ғмҡ© мҠ№мқё" (apply-approval, four-eyes) effectuates it, or "мІ нҡҢ" (withdraw) discards the draft. New-rule creation instead starts inactive (draftвҶ’publish=activate).
- **Save/apply whitelist** (changelog (16)/(18), DESIGN В§3.9.0): most direct-save actions are BANNED under this model вҖ” only a narrow whitelist may save immediately without going through the lifecycle pipeline: personal settings, event-logging-only writes, in-draft editing, permission toggles (toggles are themselves logged as a "publish" act). An explicit audit sweep (В§3.9.0) was run across the whole app to classify every save/apply call site against this whitelist вҖ” a carbon-copy build should run the equivalent classification exercise before wiring any "Save" button to a live mutation endpoint.

**Entry points**: л¬ём„ңВ·кё°лЎқл¬ј (docs) row вҶ’ lifecycle card; мқём ңмҠӨнҠё (ingest) committed-row click вҶ’ lifecycle card; к°қмІҙ нғҗмғү (explore) center-object "мғқм• мЈјкё° В· лӢЁкі„В·vN" chip.

**Reference implementation demo** (verified only in prose, not code): мЎ°м§ҒлҸ„ org-change (мҙҲм•ҲвҶ’мӮ¬м „м җкІҖвҶ’кІ°мһ¬[maker-checker]вҶ’л°ңнҡЁ[лІ„м „ N+1]вҶ’[нҸҗм§Җ мӢң]м •мӮ°[м „ліҙ/м „м Ғ/нҸ¬м§Җм…ҳ/мҪ”мҠӨнҠём„јн„°/мһҗмӮ°/кёүм—¬/4лҢҖліҙн—ҳ/нҮҙм§Ғ м°ёмЎ°л¬ҙкІ°м„ұ]вҶ’ліҙкҙҖ), `orgChange` state machine + modal entered via лІ•мқё м№ҙл“ң "мЎ°м§Ғ к°ңнҺё кІ°мһ¬".

## 12. Token grammar (`tokenParse`/`tokenRender`/`tokenPick` + PBAC gating) вҖ” POST-SNAPSHOT

Per AGENTS В§3 + DESIGN В§4.7.7 (verbatim rules below), changelog 2026-07-04(3)(g) [initial build] вҶ’ 2026-07-08(2) [`msgParts` upgrade to auto-link object codes] вҶ’ 2026-07-08(9) [critical bugfix: `tokenRender` must return an ARRAY, not a React element, or it crashes every `renderVals()` call when mapped].

Every text input across the app (кё°м•Ҳ/н•  мқј/л©”мӢ м Җ/л©”мқј/мҪ”л©ҳнҠё) shares ONE composer grammar:
- `@name` вҶ’ person/dept/entity **mention**, triggers a **notification** to the target.
- `#term` вҶ’ object **link**, no notification, inserts a clickable reference chip.
- `!CODE-NNNN` вҶ’ direct object-code link (e.g. `!AP-3121`, `!WO-2643`) вҖ” immediate link by code, no search step.
- Typed input opens a filtered dropdown (viewport-clamped, auto-flips above/below to avoid clipping).
- **PBAC-gated candidates**: the dropdown and any inserted chip only ever include objects the CURRENT USER can access вҖ” unauthorized objects are absent from suggestions (deny-by-omission) and even a well-formed `!CODE` for an object the user can't see will NOT resolve to a link.
- **Plain-text non-interference** (important, precisely specified in DESIGN.md): `@` only triggers after whitespace/line-start followed by a Korean/Latin name pattern (so `user@example.com` never triggers, nor does an already-formed `@x.com`); `#` only triggers when followed by a real search term at a word boundary (bare `#23` does not trigger, or reverts to plain text if no candidates match); `!` only triggers on the `!CODE-NNNN` shape (`!!` and ordinary punctuation are ignored). Escape, a space, or no match at any point reverts silently to plain text вҖ” the raw typed text is preserved, never mangled.
- **Explicit confirmation only**: a dropdown candidate is inserted ONLY by click or Tab вҖ” Enter/space do NOT auto-select, specifically so normal prose typing is never hijacked.
- Methods (names only, from AGENTS В§3, not read from source): `tokenParse`/`tokenRender` (display), `tokenType`/`tokenKey`/`tokenPick` (composer autocomplete), `tokenCatalog`/`tokenVisible` (the PBAC candidate-gating pair).
- **Full traceability extension** (changelog 2026-07-08(2), DESIGN В§4.7.10): `msgParts` (the message-body renderer, VERIFIED to exist in-snapshot at line 5854 but in a pre-token-grammar form вҖ” see below) was upgraded to auto-link not just `@mentions` but also bare object codes (`AP/WO/AT/CS/JL/PS/IN/DX/Bid/C/R-` prefixes) appearing in plain message text, routing clicks through `objectLinkGo` (itemsвҶ’open the panel; graph-known codesвҶ’re-center `explore`; unknownвҶ’fallback to a `docs` search). **VERIFIED-IN-SNAPSHOT partial precursor**: `msgParts(text)` (5854-5861) exists in the Jul-4 file and already does `@mention` splitting for message bubbles вҖ” the object-code auto-linking is confirmed POST-SNAPSHOT per the changelog wording ("м—…к·ёл Ҳмқҙл“ң"/upgrade), so a carbon copy should treat `@mention`-only linking as the verified baseline and code-auto-linking as the target enhancement.

DESIGN В§5's "мҷ„м „ м¶”м Ғм„ұ" (full traceability) principle generalizes this further: any billing/claim-type кё°м•Ҳ (approval draft) must funnel through STRUCTURED fields (target-person picker, evidence attachment, an automatic duplicate-claim check against existing AP- records, linked-task picker) rather than accepting free prose вҖ” changelog 2026-07-08(2)(b) cites AP-3114 (нҠ№мҲҳкІҖм§„л№„, special health-checkup expense claim) as the reference implementation of this structured-block pattern, including a "мң мӮ¬ кұҙ" (similar-case) duplicate-claim warning chip.
