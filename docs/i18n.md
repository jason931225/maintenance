# i18n Architecture

The UI is Korean-only today, but each client keeps user-facing copy in resource modules: web uses `web/src/i18n/ko.ts`, Android uses `android/app/src/main/res/values/strings.xml` with Compose `stringResource`, and iOS uses `ios/Sources/MaintenanceFieldApp/Resources/ko.lproj/Localizable.strings` with SwiftUI localized keys. Runtime data such as request numbers, customer names, equipment models, and mechanic names is not localized copy and can remain dynamic text.

`node scripts/check-i18n.mjs` runs all three enforcement checks. The web check rejects Hangul literals outside the i18n and test directories, Android rejects hardcoded Compose UI literals and missing `R.string` keys, and iOS rejects Hangul literals plus SwiftUI/localized key references that are absent from `Localizable.strings`.

Adding another locale would require translating the resource files, adding the matching Android `values-<locale>` and iOS `<locale>.lproj` bundles, and changing the web i18n module from a single `ko` export to locale selection. The current checks prove the structure is locale-ready; they do not translate content or validate translation quality.
