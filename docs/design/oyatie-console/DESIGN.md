# Acme Group 콘솔 — DESIGN.md
> 이 문서는 콘솔의 설계 헌장이다. 모든 화면·컴포넌트·텍스트·인터랙션은 아래 원칙에 대해 **항상 audit** 되어야 한다.

## 1. 핵심 철학 — 온톨로지 우선 (Palantir 벤치마크)
- 화면은 "페이지"가 아니라 **개체(Object)들의 뷰**다. 모든 명사(사람, 팀, 법인, 계약, 공고, 지원자, 업무, 날짜, 카드)는 개체이며, 개체는 어디서든 **참조·연결·이동·추적** 가능해야 한다.
- 개체 간 관계는 단순 릴레이션이 아니라 **업무 흐름 그 자체**다. UI는 관계를 따라 이동하는 경로(클릭/드래그/칩)를 반드시 제공한다.
- 목표: 사용자가 말하지 않아도 연결이 이미 되어 있는 상태. 기록은 세세하게, 분석은 쉽게.
- **온톨로지의 3계층**:
  - **의미(Semantic) 계층** — 개체가 *무엇인가*: 타입·속성·관계 (사람·조직·문서·정책의 명사 온톨로지). §2 개체 카탈로그, §4.5 카테고리·PBAC resource.
  - **동작(Kinetic) 계층** — *무슨 일이 일어나는가*: 액션·이벤트·상태 전이 (결재·열람·무단결근·수명주기 전이의 동사 온톨로지). §2 "액션·이벤트도 개체", §5 이벤트 스트림.
  - **역학(Dynamic) 계층** — *시간에 따라 어떻게 반응·변화하는가*: 규칙·정책(Cedar)·워크플로·파생 분석·시뮬레이션이 개체·이벤트에 반응해 값을 계산·전파하는 살아있는 계층. §4.5 PBAC, §4.6 no-code 정책/워크플로, §5 파생 수치·역참조.
  - 세 계층은 분리된 게 아니라 한 개체에 겹쳐 있다: 하나의 개체는 의미(정체성)·동작(생애 이벤트)·역학(적용 정책·파생 지표)을 동시에 갖는다. UI/기능은 항상 이 세 계층을 함께 노출·연결한다.

## 2. 개체 카탈로그
**구현됨**
- 조직: 그룹 전체 › 법인 › 사업장 › 팀 (조직도·근무 현황 drill, 개체 카드, @멘션 범위)
- 사람: 직원(인사 카드), 지원자, 담당/기안자
- 업무: 결재 `AP-`, 배차·정비 `WO-`, 회신 `CS-`, 근태 예외 `AT-` — 행=개체 (모달, 패널 스냅, 드래그 참조 토큰)
- 급여: 회차(파이프라인), 예외, 명부 행
- 근태: 일별 기록, 주52h, 월 마감(게이트)
- 채용: 공고(충원·파이프라인), 지원자(단계/보류/보충/탈락)
- 할 일: 범위(@사람·팀·사업장·법인) + 업무 링크 칩
- 워크스페이스: 카드 자체가 개체 (이동/크기/최소화/참조 창/분할/프리셋, 사용자별 저장)

- 커뮤니케이션·기록: 공지, 메일, 메시지 스레드 — 전부 개체 (알림은 개체를 가리키는 포인터). 할 일 = 범위+링크를 가진 개체 ✓
- **모든 결재물·기록물은 예외 없이 개체다**: 기안(상신 전 초안 — 상신 시 `AP-` 발급), 접수건(`IN-`), 공지(게시 개체), 업무일지(`JL-` = 일자×현장×작성자, 근태·정비 개체와 상호 링크). 신설되는 어떤 문서·요청·기록도 코드 발급 + 참조 토큰 + 카드 + 감사 문법을 기본 장착하고 태어난다.

- **액션·이벤트도 개체다** (Palantir 이벤트 온톨로지): 사람·조직·문서 같은 명사형 개체뿐 아니라 **행위·사건·현상**도 전부 개체다 — 플랫폼 내 액션(결재·열람·수정·상신·종결), 근태 사건(무단결근·지각·연장), 직원 생애 이벤트(결혼·출산·경조), 회사 사건(계약 체결·감사·사고), 고객/현장 사건 등. 각 이벤트 개체는 `(타입, actor, timestamp, 대상·연결 개체들, 사유, 결과)` 를 갖고 다른 개체와 링크되어 타임라인·관계 그래프·분석의 재료가 된다. (§5 이벤트 기록과 동일 구조 — 감사 로그가 곧 이벤트 개체 스트림이다.)

**계획 (백로그 순서)**

- **종결(finalization) 규칙** — 최종승인 ≠ 종결: 결재선 최종승인(또는 거절) 후에도 문서는 열려 있고, **기안자(작성자)가 결과를 확인한 뒤 직접 종결**한다(작성자 없는 자동생성 문서는 **담당자**가 종결). 결재 상태 변동(상신·단계 승인·반려·거부·종결)마다 **참여 전원에게 알림**. 종결된 문서만 상신함/결재함을 떠나 **문서함**으로 이동하고, 종결 전에는 상신함(작성자)·결재함(내 결재 완료·종결 대기)에 남는다. **상위 관리자는 Cedar 정책으로 종결을 대행·override** 할 수 있다(작성자 부재·장기 방치·감사 필요 시) — 단 override 종결은 별도 사유·감사 기록을 남긴다. **Cedar로 인가된 특정 인원은 최종승인·종결된 건도 반려·거부**할 수 있으며(예: 감사·컴플라이언스·CEO), 이 경우도 결재라인 전원에게 알림 + 사유·감사 기록이 남는다.
1. **계약/입찰 `C-`** → 2. **인원편성(포지션)** = 사업장 × 직무 × 직책 × TO → 3. **정책 프리셋**: 근무시간·휴게·근무조건·연차·수당·기타여비. 상속 체인: 법인 기본 ← 사업장 ← 직무 ← 포지션 오버라이드. 프리셋은 "설정"이 아니라 재사용 가능한 개체.
4. 공고는 **포지션 개체에서 원클릭 생성** (조건 자동 채움) → 5. 입사 확정 시 **직원 개체 자동 생성 + 타임테이블 자동 연결** (포지션 정책 상속) → 6. 근태 데이터 자동 적용 → 7. **연장근로 신청도 개체** (결재 체인 통과 시 근태·급여에 자동 반영) → 8. 급여 계산 → 인건비 분석 → **계약 수익성으로 환류**.

## 3. 관계 체인 (표준 흐름)
```
입찰/계약(C-) → 인원편성(포지션) → 정책 프리셋(근무·휴게·연차·수당·여비)
   → 모집 공고 → 지원자 → 입사 확정
   → 직원 + 타임테이블(자동) → 근태 기록(자동) ⇄ 연장근로 신청(AP- 결재)
   → 근태 마감(게이트) → 급여 회차 → 이체 → 인건비 분석 → 계약 수익성
```
- 어느 지점에서든 상·하류 개체로 한 번의 클릭으로 이동 가능해야 한다.
- 게이트(마감, 결재)는 파이프라인 스텝으로 시각화하고, **다음 행동 CTA는 항상 하나**만 노출.

## 3.9 개체 생애주기 표준 (Draft → Archive) — 전 개체 공통
정책 생애주기를 표준으로, **대부분의 업무 개체가 동일 파이프라인을 따른다**:
**초안(Draft) → 셀프 체크리스트(attestation) → 상신·결재(Submit) → 피어 체크리스트(four-eyes) → 승인(Approve — 실효 대기 effective-dating 또는 즉시 활성) → 도입·활성(Active/In-force · 민감=passkey 게시) → 철회·변경·개정 예고(Notice) → 철회·변경·개정(Revision = 새 버전 sandbox) → 보관(Archive) → 폐기(Dispose · 보존기한 만료).**
- 불변식: 모든 단계 전이 = **감사 이벤트** · **버전 관리 + 롤백**(비파괴) · **PBAC 게이트** · 상태 = **일관된 칩/스텝퍼**(§4.7 게이트=파이프라인 스텝, 단일 CTA) · 되돌리기·시뮬레이션 지원.
- 적용: 계약(C-)·포지션/정책·공고·지원자·평가·결재(AP-)·문서·기록물(IN-)·인제스트(DX-)·워크플로/자동화 규칙·예약 작업·급여 회차·증거(EvidenceRecord)·오피스 문서 버전·개체 타입(OT-). (열람·메시지 등 순수 이벤트성 개체는 생성=종결로 축약 가능.)
- **신규 기능은 이 생애주기의 해당 단계를 명시하고** 상·하류(개체 체인)와 연결한다. 기존 화면은 점진 retrofit(TODO 스윕).

### 3.9.0 저장·적용 버튼 audit (2026-07-08 directive — 직행 저장 금지)
모든 「저장」·「적용」·「확인」 버튼/함수는 위 파이프라인의 **어느 단계인지 명시**되어야 한다. 무언가가 곧바로 save/apply로 직행하면 그 자체가 설계 위반 신호 — 기업 운영·결재 생애주기에 비추어 재검토한다.
- **직행 허용 화이트리스트(예외)**: ① 개인 워크스페이스 설정(카드 배치·열 폭·테마 — 본인 소유) ② 순수 이벤트성 개체(메시지 전송·코멘트·열람 — 생성=종결) ③ **초안 저장**(초안 단계 내 편집 — 라이브 미접촉) ④ 명시적 권한 위임 하의 운영 조작(활성/일시중지 토글 = 게시 단계 권한 + 감사). 이 외의 직행 저장 발견 시: 스테이징(개정 대기 → 적용 승인)으로 전환하거나 사용자에게 적정성을 확인한다.
- **개정 스테이징 패턴**: 활성 개체의 편집 저장 = 즉시 반영이 아니라 **개정 대기(pendingRev)** — 활성 버전은 유지되고, 「적용 승인」(four-eyes·감사)으로 v+1 발효, 「파기」로 철회. (워크플로·예약 작업 = 참조 구현)
- **전수 audit 판정 (2026-07-08 스윕)**: ① 결재 행위(연차 승인 lvDecide·결재 패널 panelDecide·오퍼 제안) = 파이프라인 **결재 단계의 정당 행위**(권한+감사) — 적합 ② 분석 구조물(관계 연결·시리즈 승격·인스턴스 편입·인사이트) = 화이트리스트(감사 필수) ③ 활성/중지 토글(워크플로·예약) = 게시 권한 행위(감사 — schToggle 감사 추가) ④ **조직도 인라인 편집 = 제안(sandbox)** — orgDirty 카운터+「제안 변경 N건 · 개편 결재」 warn 배너로 §3.9.2 결재 확정 경로 연결(참조 구현) ⑤ Cedar polSave = 초안(시행은 상신·승인 후) 적합 ⑥ 인제스트 적재 = 검토 단계 진입 적합 ⑦ 메일 발송 = egress 게이트 적합.

### 3.9.1 거버넌스 메커니즘 (생성·변경·폐지 공통)
- **미래 발효일(effective-dating)**: 모든 변경은 발효일을 갖고, 초안은 라이브를 건드리지 않는 **제안 버전(sandbox)**으로 작성 — 과거 상태 재구성(as-of)·미래 계획 가능. (벤치마크: Workday staffing/org events, SAP SuccessFactors·Oracle HCM plan versions.)
- **영향 분석 / 사전점검(pre-check)**: 발효·폐지 전 자동 의존성·규정 스캔 — 영향 인원·보고라인·포지션·공고·코스트센터/예산·진행 중 결재·급여 연동·통제범위(span-of-control)·고아(orphan)/순환 보고 탐지. **차단(blocker) vs 경고** 구분.
- **maker-checker · 직무분리(SoD, SOX)**: 기안자 ≠ 승인자. 승인 매트릭스·전결규정(DoA) — HR→재무(비용)→법무(법인·노무)→임원. 법인·민감 = passkey.
- **참조 무결성 게이트**: 하위·의존 개체가 정산되지 않으면 폐지 불가.
- **변경 동결 창(freeze window)**: 급여 마감·회계 결산 중 조직 변경 제한.
- **보관 = 삭제 아님**: 활성 뷰에서 숨김이되 불변 보존(감사·법적 보존·기록물 보존기한), as-of 재구성 가능. **하드 삭제 금지.**

### 3.9.2 워크드 예시 — 조직 변경(조직도)
초안(제안 조직·발효일·상세) → **사전점검**(영향 인원·보고라인·포지션·예산·노무 요건) → 상신·검토·승인(HR·재무·법무·임원 · SoD) → **게시·발효**(원자 적용·조직 버전 N+1·이전 버전 보존) → [폐지 시] **정산**(소속 직원 전보/전적·포지션 이관·코스트센터 재배정·공고 종료·자산 이관·진행 결재 종결 + **법정 정산**: 전보/전적 동의·통지 기간·근로자대표 협의·급여/4대보험/퇴직 정산) → **폐지 사전점검**(정산 완료·보존 의무 확인) → **보관**(숨김·이력 보존·법적 보존).

### 3.9.3 안티패턴 (거버넌스)
하드 삭제(이력·감사 소실) · 라이브 직접 편집(초안·승인 우회) · 직원/개체 고아화(의존 정산 누락) · 발효일 없는 즉시 반영 · 기안자=승인자(SoD 위반) · 영향 분석 없는 파괴적 작업 · 무통지 변경(영향자·관리자 미고지).

## 3.10 내부통제 · 가드레일 (Preventive Controls) — 전 액션 공통
> 목적: **무권한·미검토·오배포 사고 방지**. §3.9 생애주기와 결합. COSO·SOX ITGC·three-lines-of-defense·four-eyes 준거. **예방(gate) > 탐지(audit/alert) > 시정**. fail-closed(기본 거부).
- **① 권한·자격 게이트(least privilege)**: 액션 *시작 전* Cedar PBAC가 principal의 권한·clearance·전결(DoA) 확인. 무권한 → 액션 미노출(deny-by-omission) 또는 사유와 함께 차단. *(계약 기안 권한 없는 직원은 계약 초안 작성 자체 불가.)*
- **② 셀프 체크리스트(self-attestation)**: 상신·전이 전 필수 체크리스트 완료 강제 — 필수 항목·검토 확인·이해상충 신고. 미완료 시 제출 불가. 서명·시각 기록.
- **③ 피어/검토자 체크리스트(four-eyes)**: 기안자≠검토자, 승인 전 제2자 검토 체크리스트. 자기검토 시스템 차단.
- **④ 결재(SoD·DoA)**: §3.9.1 승인 매트릭스·전결규정.
- **⑤ 배포/egress 게이트**: 문서·계약의 외부 발송(메일 첨부·내보내기·공유·인쇄)은 **승인/게시 상태 + 발송자 내보내기 권한 + 분류 허용**일 때만. 초안·미검토·무권한 → 차단(§13 DLP + 상태 게이트). *(미검토 계약 외부 발송 차단.)*
- **⑥ 탐지 통제**: 무권한 시도·비승인 문서 외부발송·SoD 위반 시도 = 감사(anomaly) + 컴플라이언스 알림.
- **하드 스톱 vs 권고**: 권한·SoD·egress=차단(blocker); 모범관행=경고(advisory). override=사유+상위 승인+감사.
- **원칙**: 최소권한·직무분리(four-eyes)·maker-checker·심층방어·fail-closed·자기승인 금지·전 통제 attestable·감사.
- 적용: 메일·문서·import/export·결재·워크플로·자동화·예약작업·정책·데이터 인제스트·복리후생·채용 등 모든 create/edit/advance/publish/send/export 액션.

### 3.10.1 사고 예시 매핑 — "미검토 계약 외부 발송"
사고: 직원 A가 무권한으로 계약 초안 작성 → 미검토 상태로 고객에게 발송. 가드레일 적용: ① 계약 기안 권한 없음 → 초안 작성 액션 미노출/차단 · ② 상신 시 셀프 체크리스트 · ③ 피어 검토(four-eyes) · ④ 결재(SoD) · ⑤ 발송 시 egress 게이트(계약이 승인/게시 + 발송자 권한 아니면 첨부·발송 차단) · ⑥ 무권한 초안·비승인 발송 시도 감사+알림.

## 4. UI/UX 불변식 — audit 체크리스트
매 컴포넌트·텍스트 추가/수정 시 확인:
1. **연결성**: 이 요소의 명사들은 클릭(카드/모달) 또는 드래그(참조 토큰)가 되는가? 안 되면 왜?
2. **기록**: 이 행동은 (누가·언제·왜·어떤 개체와) 4요소로 기록된다는 표시가 있는가? (감사 로그 카피, 토스트에 결과+경로 명시)
3. **예외 우선**: 정상은 조용히, 예외만 색/칩으로. 0은 숨기거나 `—`.
4. **가독 하한**: 텍스트가 읽을 수 없을 만큼 줄어드는 경로가 없어야 함 — 열 최소폭, 버튼 자동 아이콘 축소, ellipsis, 하단 fade+끝 여백.
5. **정렬**: 표는 모든 행이 같은 트랙 공식(px+fr)을 공유. per-row `max-content` 금지(행끼리 어긋남).
6. **뷰포트**: 기본 배치는 한 화면에 맞춤(비례 배분), 넘치면 페인 내부 스크롤 + overscroll 차단. 크기 조절은 뷰포트 밖으로 못 나감(단, 참조 창은 90px 손잡이 보장 하에 부분 이탈 허용).
7. **워크스페이스 소유권**: 배치·열 폭·참조 창·트레이는 사용자별 저장, "기본 배치"로 항상 복귀 가능.
8. **텍스트 규율**: 기능 없는 설명문 금지. 상태=칩, 수치·시간·코드=모노, 단위와 기준(전월/기한) 명시.
9. **매그넷 UX**: 이동/크기 조절은 그리드 틱(8~16px)+가장자리 자석, 프리셋 스톱(50/63/74%).
10. **빈 상태**: 이유 + 다음 행동을 한 줄로 (예: "근태 마감 후 계산 가능 → 근태로 이동").
11. **화면 효율**: 세로 공간은 작업 영역(목록·카드)에 우선 배분한다. 헤더·KPI·상태 스트립은 **컴팩트 1행 스탯 바**(라벨+값 인라인)로 — 큰 숫자 카드가 첫 화면 상단의 상당 부분을 점유하게 두지 않는다. 같은 패턴이 다른 화면에 있으면 함께 압축한다(§4.7 전파).
12. **설명용 UI 금지**: 기능이 없는 순수 설명·라벨·배지·"안내 카드"를 넣지 않는다. UI는 스스로 설명되어야 한다. **구체 금지**: 제목 밑 서브타이틀, 기술스택/프로토콜 캡션(예: "mox · IMAP4/SMTP"), "이 열람은 감사됨" 류 메타 안내, 칩·아이콘이 이미 보여주는 것을 반복하는 산문. 상태는 칩으로, **실제 액션을 유발하는 문구(경고·에러)만** 허용. 상태·능력은 실제 동작하는 요소(버튼·값·상호작용)로 드러내고, 배경 원칙(모바일 파리티, 감사 정책 등)은 DESIGN.md에만 기록한다.
13. **파일은 경계 포맷, 개체가 1급** (2026-07-08 directive): 콘솔 안에서 이력서·계약·전표·명세 등 모든 업무 내용은 **구조화된 콘솔 개체(필드·관계·생애주기)가 primary**다. xlsx/pdf/docx는 (a) 유입 시 인제스트로 구조화되고 원본은 provenance/증거(WORM)로 링크, (b) 유출 시 export 산출물(egress 게이트) — backward/external compatibility 전용. 화면에서 「OO.pdf 열람」이 primary 동선이면 안 되고, 구조화 뷰가 primary·원본 파일은 「원본」 보조 링크로 강등한다. (예: 지원자 카드 = 구조화 프로필 + 원본 PDF provenance 칩)
14. **모듈 = 개체 서피스, 텍스트 래퍼 금지** (2026-07-08 directive): 모듈의 모든 행(entry)은 typed 개체이고 모듈은 개체 타입이다 — 컴플라이언스 의무(CP-)·규제(RG-)·전표(VC-)·정책·기록물·결재·감사 이벤트·분석(AN-)·예측(FC-) 전부. 모든 개체는 ① **온톨로지 그래프에 합류**(kv·링크의 코드 = 자동 엣지), ② **개체 카드 3계층** — 의미(속성·관계) + 동작(생애주기 스텝퍼·버전) + 역학(작용 자동화·예약·정책·시리즈), ③ **거버넌스 CRUD** — 생성=기안·등재·인제스트, 열람=감사, 수정=개정(v+1 sandbox), 삭제=보관·폐기 게이트(하드삭제 금지), ④ **관계 그리기** — 코드 입력·드래그 드롭으로 엣지 생성(감사·해제 가능). 표시 형식은 데이터 성격에 맞게(목록·타임라인·그래프·매트릭스) 선택한다.
    **모듈 3계층 audit** (2026-07-08 directive): 새 모듈·화면은 세 계층 노출을 점검한다 — 의미(이 화면의 명사는 속성·관계를 가진 개체인가), 동작(상태 전이·이벤트가 생애주기·감사로 남는가), 역학(작용하는 자동화·정책·시리즈·파생 지표가 표면에 보이는가). 하나라도 빠지면 §4-14 위반.
15. **지속 개체(시리즈) ↔ 인스턴스** (2026-07-08 directive): 반복되는 지불·청구·이벤트·액션·패턴(임대료 월 청구, 장비의 정비 오더들, 분기 검진, 월 급여 회차, 대근비)은 **부모 시리즈 개체(SR-)** 로 캡처한다 — 주기(rule)·인스턴스 이력·추세·다음 예정을 갖고, 각 인스턴스(월별 AP-, WO-)는 시리즈에 링크. 과거 맥락이 소실되지 않고, 변화 비교·미래 예측이 수동 발굴 없이 가능해야 한다. 인스턴스 카드에는 시리즈 미니 타임라인, 시리즈 카드에는 전체 이력+예정.
16. **규제·파라미터 = 실행 가능한 원장 개체** (2026-07-08 directive): 법령·규제 값(최저임금 고시, 주52h 상한, 검진 주기)은 텍스트가 아니라 **참조·플래깅·시뮬레이션·파생 액션을 구동하는 개체(RG-)** 다 — 영향 평가(대상 인원·비용·마진), 조정 기안 프리필, 발효일 자동화(예약 작업), 감시 모델(FC-) 연결. 기록·결정·액션이 모두 개체로 남는 actionable ledger.
17. **온톨로지 스키마 = 개체** (2026-07-08 directive): 개체 **타입 정의(OT-)·관계 타입 자체가 §3.9 생애주기를 따르는 개체**다 — 제안(초안 sandbox) → 데이터 거버넌스·임원 검토(SoD) → 활성(스키마 버전 v+1) → 개정 → 보관(deprecate). **보관 게이트 = 참조 인스턴스 마이그레이션 + 작용 자동화·정책 재바인딩** (참조 무결성 — 하드삭제 금지, 기존 데이터의 as-of 재구성 보존). 타입 레지스트리는 그래프 범례가 곧 진입점(범례 칩=타입 카드, 관리 주체·버전·활성 개체 수)이고, 인스턴스 노드는 소스 개체의 생애주기를 따라 그래프에 편입·퇴장한다. (Foundry Ontology Manager 벤치마크)
   **그래프 저작(authoring)**: ① 새 개체 = 탐색 패널 「+ 새 개체」(타입 선택+이름) → **초안(OB-)으로 생성**되어 상신·검토·승인 생애주기를 거쳐 활성 ② 새 부모 = 개체 카드 「시리즈 승격」 → 지속 개체(SR-) 생성 + 첫 인스턴스 편입, 이후 시리즈 카드에 코드 입력·드롭 = **인스턴스 편입**(추세 재계산) ③ 새 관계 = 개체 카드 입력·드롭 + **탐색 중심 개체에 행·노드 드롭** = 감사되는 엣지(해제 가능) — 그래프 노드는 드래그 소스(참조 토큰과 동일 payload).
18. **컴포넌트 재사용·표준화** (2026-07-09 directive): 새 UI는 **기존 프리미티브 확장이 기본**, 새 패턴 발명은 예외(헌장 등재 필수). 표준 인벤토리 — ① **모듈 서피스 cfg 문법**: `rows(code·c1~c3·st·tone·kv·links·acts)` + 확장 필드 `stats·lanes(칸반)·stock(재고 바)·tl(타임라인)·prog(진행 바)·ctl(통제→증거)` — 새 목록 화면은 MOD_SCREENS 항목 추가로(커스텀 템플릿 금지) ② **개체 카드 = lcOpen 단일 모달**(스텝퍼·속성·이력·결재선·게이트·역학 칩·관계 그리기 — 새 개체 타입은 lcSeed 분기 추가) ③ **톤 = TONE(t) 단일 팔레트**([bg,bd,tx,solid] — 인라인 삼항 중복 금지) ④ 공용 문법: 세그먼트 컨트롤(칩 그룹+ink 활성)·상태 칩·코드 모노 칩·드래그 참조 토큰(`[코드]` payload)·드롭 존 하이라이트·Esc 닫기 체인·패널 승격·모달 골격(헤더 칩+바디+CTA 푸터). 위반 신호 = 같은 모양을 두 번 손으로 그리는 것.

19. **필드 타입 규율 (typed fields · 2026-07-09 directive)**: 개체의 속성·조건·자격·사유는 **curated enum 또는 구조화 타입**(코드 참조·금액·일자·발효일·기간·수량·백분율)이 기본이다 — 자유 텍스트는 enum 선택에 부속된 상세 보완만. 분석·집계·정책 트리거·자동화 조건은 enum/구조 필드에서만 가능하므로, 텍스트로 남은 필드는 그 개체가 역학 계층에 참여할 수 없다는 신호다. 적용 예: 신청 사유(경비=교통·식대…)·탈락 사유 5종·복리후생 tier/Cedar 조건·공고 자격 요건(req[])·고용 형태(emp)·재고용/평점·SLA. audit 질문: "이 텍스트/숫자는 enum·코드·일자·금액 타입이어야 하는가, 진짜 산문인가?" 신규 필드는 타입 판정 없이 텍스트로 추가하지 않는다.
20. **모든 개체 표면 = 드래그 소스 (2026-07-09 directive)**: 개체를 표상하는 모든 칩·행·코드 라벨은 draggable — 표준 참조 토큰 payload `[코드 제목]`, 단일 정의 `objDrag(code,title)`. 드롭 타깃(컴포저·할 일·관계 드롭 존·시리즈 편입)은 payload의 코드를 파싱한다. **모듈 자체도 개체(MD- 코드)**: nav 항목이 드래그 소스 — 메신저·메일·할 일에 모듈 참조를 첨부해 "어디서 찾는지" 안내가 링크로 전달되고, MD- 토큰 클릭=해당 모듈 이동(비인가 모듈은 PBAC 차단). 같은 모양의 개체 칩을 새로 그리면 드래그 소스 여부를 함께 판정한다.

## 4.6 안티패턴 — 절대 금지 (반복 위반 주의)
- **설명 캡션·서브텍스트**: 제목 밑 부제, 기술스택/프로토콜 캡션(예: "mox · IMAP4/SMTP"), "이 열람은 감사됨" 류 메타 안내, 칩·아이콘이 이미 보여주는 것을 반복하는 산문 → 삭제. 상태=칩, **실제 액션 유발 문구(경고·에러)만** 허용.
- **기능 없는 텍스트·데이터 슬롭**: 기능(클릭/드래그/필터) 없는 라벨·배지·숫자·통계·아이콘 금지 (§4-1 연결성).
- **AI 슬롭 비주얼**: 그라디언트 배경·이모지(브랜드 제외)·둥근 컨테이너+좌측 컬러 보더·남용 폰트(Inter/Roboto/Arial) 금지. 색은 토큰·브랜드에서만.
- **큰 숫자 KPI 카드** → 컴팩트 1행 스탯 바(§4-11). **채우기 콘텐츠**(더미 섹션·여백 채우기) 금지.

## 4.5 접근 제어 — 콘솔 전역 Cedar 기반 PBAC
- **콘솔 자체가 Cedar PBAC 위에서 동작한다**: 화면(모듈), 카드, 테이블 행, 액션 버튼, 검색·팔레트 결과, 드래그 참조, 알림까지 — 렌더링되는 모든 것이 정책 평가(`permit(principal, action, resource) when {…}`)의 결과다. 카드 섹션은 그중 가장 세분화된 resource 단위일 뿐이다.
- **principal** = 직책·직급·직무·대상(본인/직속/담당) 속성, **resource** = 개체의 정보 카테고리(카드 섹션) 단위 attribute, **action** = view / edit / export.
- 모든 개체 카드(직원·법인·계약·공고 등)의 정보는 **카테고리 = 정책 적용 단위**로 묶는다. 카테고리마다 접근 칩(공개 범위)을 상시 표시하고, 민감 카테고리는 기본 접힘 + "열람 — 기록 남음" 게이트.
- **권한이 있어도 열람은 반드시 기록**: `(actor, target object, category, timestamp)` — 감사 로그 개체로 남기고 UI에 "열람 기록됨" 배지.
- 정책 준비 전에는 칩·게이트를 UI 계약으로 먼저 배치("권한·정책 연동 예정" 라벨), 정책 도입 시 칩이 실제 Cedar evaluation 결과로 전환된다.

- **비밀 구역 (Covert clearance)**: 그룹 최고관리자(CEO)가 지정한 **비밀인가 인원**만 접근하는 최상위 등급. 원칙은 *deny-by-omission* — 인가되지 않은 principal에게는 섹션·메뉴·검색 결과·칩이 **아예 렌더링되지 않는다** (잠금 표시도 금지: 비밀의 존재 자체가 정보). 인가 인원 명단과 clearance role 자체도 비밀 resource로 취급해 동일 규칙 적용. 인가자의 열람·수정은 CEO만 조회 가능한 별도 감사 스트림에 기록.

- **"전체"는 principal-상대적이다**: 스코프 선택기의 "그룹 전체"는 문자 그대로의 전체가 아니라 **그 사용자가 인가된 법인들의 합집합**이다. 그룹 단위 접근권이 있어도 그룹의 모든 정보에 접근하는 것은 아니다. 모든 집계(KPI, 지급 총액, 출근율, drill 루트, 검색·팔레트 결과, 배지 카운트)는 인가 범위 안에서만 계산·표시되고, 접근 없는 법인은 스코프 목록·합계·건수 어디에도 나타나지 않는다. 법인의 "존재" 노출 여부는 개체별 정책을 따른다(비공개 법인은 deny-by-omission).

- **데이터 유출 방지 — 정직한 위협 모델(보안 극장 금지)**: 순수 웹앱만으로 스크린샷·OS 복사·인쇄를 원천 차단할 수 없다(**WASM도 동일 샌드박스** — OS 권한 없음). 콘솔은 **억제(user-select·앱-내부 클립보드·canvas 렌더·동적 워터마크·blur-on-blur·print 인터셉트) + 추적(모든 열람/복사/내보내기/인쇄를 감사·워터마크로 유출 추적) + 게이트(covert/민감은 관리 기기·사내망·passkey에서만 렌더 = deny-by-omission)**를 구현하고, **완전 방지는 엔터프라이즈 계층(엔터프라이즈 브라우저 Edge for Business+Purview·Netskope·Island·Prisma / VDI·RBI / 엔드포인트 DLP·MDM)을 배포 요건**으로 둔다. 어떤 UI도 "완전 차단"을 약속하지 않는다. (HANDOFF §13)

## 4.6 비개발자 우선 (No-code first)
- **Cedar 정책도, 워크플로도 no-code 비주얼 캔버스 에디터가 기본이다.** 대상 사용자는 비개발자(운영·인사·관리자). 코드·JSON·정책 문법을 UI에 노출하지 않는다 — 자연어 라벨("경비팀 팀장은 소속 팀원의 근태를 볼 수 있다")과 블록/노드 캔버스로 표현.
- 정책·워크플로 편집에는 반드시 **미리보기 + 시뮬레이션**("이 정책이면 누가 무엇을 보게 되는가", "이 흐름이면 어떤 개체가 어떻게 움직이는가")과 **되돌리기/버전 기록**을 제공.
- 콘솔의 모든 구성·유지보수 — 개인 화면(카드 배치·열 폭·참조 창), 카테고리·프리셋, 정책, 자동화 — 는 비개발자가 UI 조작만으로 수행할 수 있어야 한다. 이미 구현된 워크스페이스 패턴(드래그 배치, 프리셋, 매그넷)이 기준선이며, 정책·워크플로 편집기도 같은 문법을 따른다.
- audit 질문 추가: "이 기능을 비개발자가 문서 없이 스스로 발견하고 조작할 수 있는가? 실수하면 스스로 되돌릴 수 있는가?"

## 4.7 패턴 전파 규칙 — 한 번 확립된 문법은 전체에 적용된다
- 한 컴포넌트에 도입된 UI/UX 패턴은 그 컴포넌트만의 기능이 아니라 콘솔의 **문법(grammar)**이다. 같은 형태의 UI를 만들거나 발견하면 적합성을 판단해 **자동 전파**하고, 적용이 모호한 경우에만 한 번 확인 후 전체 적용한다. 사용자가 컴포넌트마다 같은 요청을 반복하게 하지 않는다.
- **확립된 문법 카탈로그** (지금까지의 directive 정리):
  1. **목록·테이블**: J/K/Enter 키보드 탐색+선택 하이라이트 · 열 폭 드래그(가독 하한, 8px 틱) · 다속성 검색(이름·부서·법인 등 보이는 모든 속성) · 끝 여백+overscroll 차단+하단 페이드 · 모든 행 공유 트랙 정렬(per-row max-content 금지) · 좁아지면 버튼 자동 아이콘 축소 · 예외만 칩, 0은 숨김/—.
  2. **카드·컴포넌트 — 창(window) 조작 모델 (모든 카드/패널 공통, 단일 정의)**: 4개 상태 — 기본(그리드) · **핀(분할 패널)** · **팝아웃(자유 플로트)** · **최소화(작업 트레이)**. 조작 문법:
     - **상단 바(헤더) 드래그** = 팝아웃(자유 플로트, 커서 추종·매그넷 스톱). 드래그는 헤더 밴드(상단 ~54px)에서만 시작 — 본문 스크롤·행 클릭 비간섭. 인터랙티브 요소(button·input·a·[role]) 위에서는 미발동.
     - **더블클릭(헤더)** = 핀(분할 패널). 우측 도킹 + **본문이 옆으로 재배치**(section `padding-right`로 실제 공간 확보 = 오버레이 아닌 진짜 split). 재더블클릭 = 해제.
     - **작업 트레이로 드래그** 또는 **최소화 버튼** = 트레이 칩(복원 가능).
     - **버튼 3종**(hover 컨트롤 — 카드는 `cardToolVals`, 상세 패널은 공유 헤더에 1곳 정의): **핀 / 최소화 / X(닫기=기본 배치)**. 상세 패널은 이미 pin 상태이므로 팝아웃·최소화·X.
     - 핀(플로트)은 `position:fixed`+`visibility:visible`로 hidden 화면 래퍼 탈출 → **다른 화면 이동에도 유지**(§4.7 참조 창 = 이 메커니즘). 리사이즈·사용자별 저장·기본 배치 복귀 유지.
     - **중앙 정의 원칙**: 조작 behavior는 `cardVal`/`cardToolVals`/`cardGrab`/`cardPinRight`/`cardRestoreDefault`에, 상세 패널 chrome은 단일 `<sc-for panels>` 헤더에 정의 — 변경은 한 곳에서 전체 전파(카드 콘텐츠 슬롯까지 재사용 컴포넌트화는 HANDOFF backlog).
  3. **개체**: 코드 발급(AP-/WO-/AT-/JL-…) · 드래그 참조 토큰 · **클릭 시 상세는 핀 패널로 열린다(기본)** — 화면 quadrant에 고정되어 다른 화면으로 이동해도 유지되고, 패널 안팎으로 드래그·드롭이 자연스럽다 · 링크 칩으로 상·하류 이동 · 모든 행동 감사 기록 · PBAC 카테고리.
  4. **개체 카드**: 정보 카테고리+접근권한 칩+열람 기록 게이트, 비밀 구역은 deny-by-omission.
  5. **메뉴·팝오버**: 바깥 클릭/Esc로 닫힘, 위치는 트리거 기준.
  6. **게이트·마감**: 파이프라인 스텝 시각화 + 단일 컨텍스트 CTA.
  7. **토큰 문법 (모든 입력창 공통)** — 콘솔 전 입력(기안·할 일·메신저·메일·코멘트)에서 동일:
     - `@` = 사람·부서·법인 멘션 → 대상에게 **알림 발송**.
     - `#` = 개체 링크 → **알림 없음**, 삽입된 칩은 클릭해 개체로 이동 가능.
     - `!` = 개체 코드/슬러그 직접 링크 (예: `!AP-3121`, `!WO-2643`) → 코드로 개체를 즉시 링크.
     - 입력 중 검색형 드롭다운(타이핑에 따라 후보 필터). 드롭다운은 **뷰포트를 벗어나 잘리지 않게** graceful 처리(위/아래 자동 뒤집기, 경계 클램프). 선택 시 칩/링크로 확정.
     - **PBAC 게이트**: 드롭다운 후보와 태그는 **현재 사용자가 접근 권한이 있는 개체만** 노출·삽입 가능하다(§4.5) — 권한 없는 개체는 후보에 나타나지 않고(deny-by-omission), 코드(`!`)로 직접 입력해도 링크되지 않는다.
     - **일반 텍스트 비간섭**: `example@example.com`, `#23`, `주의!!` 같은 평범한 텍스트에서는 트리거가 발동하지 않아야 한다. 발동 조건 — `@`는 앞이 공백/줄머리이고 뒤가 한글·영문 이름 패턴일 때만(이메일 로컬파트 뒤 `@`, 이미 `@x.com` 형태면 무시), `#`는 단어경계 뒤 검색어가 이어질 때만(순수 숫자 `#23`은 무시하거나 확정 전 후보 없으면 그대로 텍스트), `!`는 `!CODE-NNNN` 코드 패턴이 이어질 때만(`!!`·문장부호는 무시). 드롭다운이 뜨더라도 Esc·공백·불일치 시 즉시 평범한 텍스트로 되돌아가고, 확정하지 않으면 입력 원문이 그대로 남는다.
     - **명시적 확정만**: 드롭다운(검색)은 떠도 **클릭 또는 Tab** 으로만 후보가 선택·삽입된다. 공백·Enter는 자동 선택하지 않고 입력한 텍스트를 그대로 둔다 — 평문 입력을 방해하지 않기 위함.
  8. **역학 노출(dynamics surfacing)**: 개체 표면(그래프 패널·카드)에는 그 개체에 **작용하는 자동화·정책**을, 자동화 규칙에는 그 규칙이 만지는 **개체 체인**을 칩으로 노출 — 양방향 1클릭(explore↔auto). 자동화·정책 편집 블록은 반드시 온톨로지 개체 타입에 바인딩(자유 텍스트 블록 금지).
  9. **분석 = drill 불변식**: 집계 화면의 모든 수치·바·행은 원본 개체로 이동하는 버튼이다. 클릭 불가 숫자·장식 차트 금지(§4-12와 동치).
  10. **완전 추적성(언급=링크)**: 본문·메시지·알림 속 개체 코드·인명은 평문이 아니라 토큰 링크여야 한다. 청구·신청류 기안은 대상자 명단(개체)·증빙·**중복 청구 자동 점검**·연결 업무를 구조 필드로 유도 — 자유 산문만으로는 처리되지 않게 한다.
- **상세 보기의 기본은 핀 패널이다 · 통일 제스처**: 콘솔 전체가 한 가지 핀 모델을 쓴다 — **클릭**=상세를 우측 핀으로 열기(카드·개체 공통 기본), **드래그**=끌어서 드롭한 쿼드런트에 핀, **더블클릭**=우측 패널 고정(재더블클릭 해제). 별도 핀 버튼이나 hover 툴바는 두지 않는다. 화면 중앙을 점유하는 창(centered modal)은 기본이 아니며, 핀 패널 렌더러가 아직 없는 개체(예: 근태 예외 hrissue)에 한해 임시로 쓰고 순차적으로 핀 패널로 수렴시킨다(HANDOFF.md 참조).
- 새 컴포넌트는 이 카탈로그를 체크리스트로 통과해야 하고, 문법이 개선되면 같은 형태 전체에 **소급 적용 audit**을 돌린다. 이 카탈로그 자체도 새 directive가 나올 때마다 갱신한다.

## 4.8 사용자 범위 — 관리자만이 아닌 전 직원
- 콘솔은 관리자 도구가 아니라 **전 직원(현장직·생산직·정비직 포함) 시스템**이다. 모든 직원이 최소한 사용하는 것: 전자결재(기안·상신·본인 결재), **본인 급여 내역(셀프서비스)**, 메신저·메일·알림·주소록·게시판.
- PBAC principal이 곧 화면을 결정한다(§4.5): 일반 직원은 관리 기능·타인 민감정보 없이 **자기 개체(본인 급여·근태·결재·할 일)** 중심 뷰를 본다. "관리자 전용"은 예외이지 기본이 아니다.
- **모바일 우선 고려**: 메신저·메일·알림·주소록·전자결재는 모바일 앱에도 존재한다. 이 기능들은 좁은 폭·터치 타깃(≥44px)·오프라인/푸시를 전제로 설계하고, 데스크톱 콘솔과 **동일 개체·동일 상태**를 공유한다(뷰만 다름).
- **커뮤니케이션 모듈 승격(rail ↔ main)**: 메신저·메일·알림·공지·주소록은 평소 우측 커뮤니케이션 rail에 컴팩트하게 상주하다가, 사이드바에서 해당 모듈을 열면 **rail에서 사라지고 메인 패널에 풀뷰로 승격**된다(넓은 목록·상세 2단, 검색·필터). 다른 화면으로 이동하면 다시 rail로 복귀한다 — 같은 개체·같은 상태를 rail(요약)과 main(풀뷰)이 공유(뷰만 다름).
- 셀프서비스 원칙: 본인 데이터는 스스로 열람·신청(기안)할 수 있고, 타인/집계 데이터는 PBAC로 게이트. 본인 급여 열람은 감사 기록 대상이 아니되(본인 권리), 타인 급여는 기록.
- **개인 Inbox (수령·열람 확인)**: 본인 급여 명세를 일반화한 **개인 수신함**을 둔다 — 급여명세, 그리고 회사가 **수령 확인**을 받아야 하는 법적 문서(전자결재로 못 하면 서면으로 해야 하는 것들: 근로계약·연차촉진 통지·노무수령거부권 행사·취업규칙 변경 등)를 이 inbox로 전달한다. 각 문서는 **수령/열람 확인** 상태를 갖고(누가·언제 확인), 확인이 곧 법적 증빙이 된다.
- **열람 passkey 인증**: 민감·법적 문서 열람은 **passkey(본인확인) 인증**을 거쳐야 열린다 — 열람 자체가 본인 수령 증빙이므로. (자기 급여 등 일상 열람은 무마찰, 법적 수령확인 문서는 passkey 게이트.)
- **연차 촉진·노무수령거부권도 전자결재 개체**: 연차 사용 촉진 1차/2차 통지, 노무수령거부권 행사는 전부 전자결재(AP-)로 상신·발송되고 개인 inbox로 수령 확인된다(근로기준법 §61). 서면 대체가 필요하면 동일 개체에서 서면 출력·회수 확인으로 이어진다.

- **기안 유형별 개체 연결** (compose 시 "개체 연결"로 지정): 복리후생=지급 대상자(직원/팀/사업장/법인, 필수 — 법인 차원 담당자 기안), 경비 청구=증빙+프로젝트/업무, 지출결의=계약/거래처, 구매=자산/재고/계약, 연장근로=정비오더(WO-)/근태(AT-), 대근=대상 현장/결원 인원, 연차=근무표. 연결 개체는 참조 토큰으로 남아 상·하류 추적·분석에 쓰인다.

- **사유는 자유 텍스트가 아니라 enum**이다: 기안·연차·경비 등 모든 신청의 사유는 유형별 curated enum(예: 경비=교통·식대·숙박·소모품·접대·기타)에서 선택하고, 상세는 선택적 자유 텍스트로 보완한다. enum이어야 분석·집계·정책 트리거가 가능하다.

- **복리후생 조건은 Cedar no-code로 세부조정**: 각 제도의 적용 대상·차등은 법인별·사업장별로 다를 수 있고 개인별(입사시기 등)로도 달라진다. 조건은 자연어 규칙 블록(no-code 캔버스)으로 편집하며 principal 속성(현장·직책·직급·입사일)에 대해 평가된다.

- **복리후생 제도 = 편집 가능한 개체 + 정책 수명주기**: 각 제도는 단순 텍스트가 아니라 typed·linked 개체이며 수정 시 실제 반영된다. 정책 변경은 수명주기를 따른다 — 초안(draft) → 승인 대기(pending) → 확정(finalized, 시행일 preset) → 시행(implemented) → 폐지 예정(retiring, 폐지일 preset) → 폐지(retired). 상태 전이는 참여자 알림·감사 기록을 남기고, 차등 tier(현장·직급·직책)와 Cedar 조건에 연결된다. 이 수명주기 패턴은 다른 정책성 개체(정책 프리셋 등)에도 전파한다.

### 4-19. 재사용 초기화 · 단일 chokepoint (2026-07-09 directive)
- **보안·정책 로직은 단일 평가 지점으로만**: 화면 접근=`can("screen:view")`+`cedarScreenGuard`(componentDidUpdate — 팔레트·토큰·딥링크 전 경로 fail-closed) · 인물 가시성=`personVisible`/`peopleAllowed` · @자동완성=`mentionPeople`/`mentionScopes`. 개별 컴포저·컴포넌트에서 PEOPLE/MEMBERS 직접 조회 금지 — 새 입력·목록은 반드시 위 헬퍼를 통한다. 차단은 항상 감사(deny)로 기록 — 조용한 회귀(silent regression) 금지.
- **레이아웃 프리셋**: 모듈 테이블 그리드는 가용 폭(사이드바·rail·상세 패널 차감) 기준 풀/컴팩트 자동 전환(`modNarrow` — 컴팩트=코드·제목·상태만). **테이블 수평 스크롤 금지** — 고정 px 트랙 대신 minmax, 래퍼 min-width 금지, 압축은 ellipsis가 흡수.

### 4-20. 온톨로지 엔진 = single engine, multiple consumers (2026-07-09 directive · Palantir Foundry/Maven 벤치마크)
> **정체성**: Palantir의 파운드리·온톨로지는 AI-우선이지만, **우리는 알고리즘·기계적·프로그래매틱·no-code 우선**이다(결정적 no-AI 헌장). 그렇다고 UI·UX·기능이 뒤처져선 안 된다 — adaptable·intuitive·ergonomic을 같은 수준으로 달성한다. AI가 하던 일은 규칙·템플릿·술어 평가·시뮬레이션으로 치환한다.
- **하나의 엔진, 여러 소비자**: 개체·개체 타입·관계·분석은 모듈마다 하드코딩/하드패싱하지 않는다. 단일 레지스트리(`ONT_TYPES()`)가 **개체 타입 = typed 속성 스키마 + 관계(링크) 유형 + 액션(writeback) + 분석(파생 속성)** 을 한 곳에 정의하고, 모든 표면이 이를 **참조**한다 — 객체 탐색 그래프·타입 스키마 카드·정책(principal/resource)·워크플로 블록·모듈 서피스. 타입/속성/관계/분석을 엔진에 추가하면 전 소비자에 자동 전파(예: 타입 추가 → 정책 resource 후보 자동 노출). "같은 온톨로지를 두 번 정의하면 위반".
- **configurable = 기능적 타입 config, 무기능 텍스트 금지**: 설정·조건·파라미터는 typed·actionable 입력(enum·number·operator·field 선택)으로만. 자유 문자열 라벨을 "설정"인 척 두지 않는다. 조건은 **field · operator · value 술어**로 실제 평가되고, 시뮬레이션은 시드 표본에 술어를 **실행**해 결과를 계산한다(장식용 토스트 금지). 로직·통합·임계·라우팅·매핑은 정책/설정 개체(no-code 편집·개정 스테이징 §3.9.0)로 추출 — 코드 상수 하드코딩 금지.
- **개체 = 속성 + 관계 + 액션 + 역학 (3계층 초월)**: 그래프는 노드 간 이동을 넘어 개체를 **검사·조작**한다 — typed 속성(타입 배지), typed 관계(링크 유형·from/to), **invokable actions**(개체를 mutate하거나 컨텍스트로 이동, 전량 감사 — Palantir Actions), 파생 분석(산식). 클릭 이동만 하는 그래프는 미완.
- **엔진 편집 = no-code 거버넌스**: 타입 스키마(속성)·관계 유형·분석 정의는 타입 카드에서 편집하며, 활성 타입은 개정 스테이징(§3.9.0 — 적용 승인 four-eyes·v+1)·초안은 직행. `state.ontTypeDefs` 오버레이가 편집분을 병합(기본 정의 불변).

### 4-21. 모듈별 3문 벤치마크 리뷰 루프 (2026-07-09 directive — 상시 프로세스)
- 각 모듈·컴포넌트 구현 완료 시 반드시 3문 리뷰를 수행한다: **"Palantir라면 무엇을 더 잘할까? SAP라면? Slack이라면?"** (해당 모듈의 best-in-class로 치환 가능 — Workday·Greenhouse·Gmail 등). 답은 TODO 「벤치마크 갭 레지스터」에 환류하고, 다음 심화 슬라이스는 거기서 선정한다. 리뷰 없는 "완료" 선언 금지.

### 4-22. 다중 유스케이스 · 보편 확장성 (2026-07-09 directive)
- **모든 요소는 2개 이상의 사용자 스토리에 복무한다**: 모듈·컴포넌트·요소가 단일 용도 장식이면 위반 — 조회+생성 진입, 표시+필터 조건, 데이터+드래그 참조처럼 최소 두 스토리에 걸쳐야 존재 가치가 있다. "이 요소가 복무하는 스토리 2개를 답하지 못하면 제거하거나 확장한다."
- **보편 확장성 (add-anything)**: 사용자가 마주치는 모든 UI 요소는 그 자리에서 **새 항목을 추가할 수 있어야 한다** — 목록=새 행(도메인 액션/위저드), enum 칩=새 값 제안(타입 스키마 경유), 테이블=새 열(구성 모드), 스탯 바=새 스탯, 타입=새 속성·관계·액션·분석, 쿼리/필터=저장 가능한 프리셋. placeholder 없이 end-to-end(§HANDOFF 20 CRUD 매트릭스), 전이는 §3.9/§3.9.0 거버넌스를 통과. 벤치마크: Foundry(온톨로지 저작)·Airtable(어디서든 필드 추가)·Notion(인라인 스키마)·ServiceNow(UI policy).

### 4-23. 보편 창 모델 · 드래그 참조 후보 (2026-07-09 directive)
- **모든 모듈·카드·컴포넌트·요소는 창 모델 후보다**: 작업 트레이(최소화)·자유 오버레이(팝아웃)·모달·핀 분할 패널 — 새 표면을 만들 때 "이건 어느 창 상태를 지원하는가"를 명시하고, 합당한 상태는 기존 문법(`cardVal/cardGrab/snapTo/panels`·트레이 Dock)으로 즉시 제공한다. 지원 불가가 기본값이면 위반.
- **모든 요소는 드래그 참조 후보다**: 어떤 표면의 개체·행·칩·코드든 텍스트 입력(메신저·기안·메일·기록·업무노트·코멘트)으로 **끌어 놓으면 참조 토큰**(`[코드 제목]`)이 된다 — `objDrag` 단일 헬퍼(§4-20 드래그 소급)로 신규 표면도 등록. 받는 쪽은 토큰 문법(§4.7 @#!)이 해석. "끌 수 없는 개체 표기"는 위반.

### 4-24. 차트 정직 스케일링 (2026-07-09 directive)
- **가시성과 정직 사이 규칙**: 값 분산이 좁아 0 기준으로는 차이가 안 보이면(상대 분산 < ~1/3) **축을 절단**해 차이를 보이게 하되, **절단 사실을 반드시 명시**(「축 절단 — 기준 ₩xxx (0 아님)」 warn 칩). 분산이 충분히 크면(예: 5%↔10%) 0 기준 유지 — 왜곡 없는 쪽 우선. 모든 차트·바·스파크라인 공통. 참조 구현 = 대시보드 인건비 추이.

### 4-25. 폐루프 페이지 리뷰 프로토콜 (2026-07-10 directive — 상시 프로세스)
모든 사용자 가시 페이지에 대해 폐루프로 순회하며 다음을 묻고, 답을 레지스터에 환류하고, 픽스를 적용한 뒤 재검증한다:
1. **패턴**: 지금 이게 가장 직관적·인체공학적·best-in-class 패턴인가? 구성 가능·재현 가능·유지보수 가능한가? 더 낫게 만들 수 있는가?
2. **워크플로 커버리지**: 이 화면이 모듈의 워크플로를 — 나아가 **전사 업무 운영 워크플로**를 — 담는가? 뉘앙스·엣지 케이스(예외·부분·소급·병행)를 커버하는가?
3. **마찰 예측**: 사용자가 걸려 넘어질 지점·문제·불편을 예견하라. 시선·정보의 자연 흐름(위→아래, 좌→우, 요약→상세) UX 원칙을 지켰는가?
4. **벤치마크**: 해당 도메인 best-in-class 연구 대비 폴리시 기회 식별 (§4-21 3문과 결합).
5. **통합·재사용**: 이 모듈이 나머지 모듈과 긴밀히 연결되는가(§5 통합 원칙)? 재사용 컴포넌트를 완전히 활용하는가 — 같은 모양을 새로 그렸다면 위반(§4-18)?
6. **목업 독립성**: 이 모듈은 — 그리고 콘솔 전체는 — 스텁·목 데이터 없이 오늘 동작하는가? 판정 기준: 화면의 모든 데이터가 ⓐ state 파생(UI로 변경 가능)이거나 ⓑ UI 생성 경로가 있는 시드다. 하드코딩-전용 데이터·스텁 동작은 갭 — 발견 즉시 등재·마감. (백엔드 필요 항목은 HANDOFF 계약으로 명시 — "백엔드성"을 이유로 UI 경로 부재를 허용하지 않는다.)
7. **페르소나·유저 스토리 커버리지**: 이 모듈에 관여하는 페르소나는 누구인가(본인·팀장·전담자·관리자·외부 — ROADMAP §8 매트릭스·VIEWERS)? 타겟 청중의 유저 스토리를 빠짐없이 담는가? 각 스토리가 UI에서 **직관적 동선으로 실제 수행 가능**한가(actionable — 설명 없이 따라갈 수 있는가)? 미지원 페르소나·끊긴 동선 = 갭 등재 (view-as로 e2e 검증).
8. **레이아웃·마진**: 마진·간격이 균일하고 관례(8px 격자·gap 일관)를 따르는가? 컴포넌트·콘텐츠가 한 화면에 들어오거나, 넘치면 합당한 이유(스크롤 존·drill)가 있는가? 같은 정보를 더 잘 전달할 **다른 레이아웃·다른 표현 형식**(표↔칸반↔타임라인↔매트릭스)을 검토했는가(§4-14-④)?

### 4-26. SLO ≠ SLA (2026-07-10 directive)
- **SLA = 계약상 대외 약정**(고객 계약 C-·현장 서비스 수준 — 위반=페널티·계약 개체 귀속·egress급 심각도) · **SLO = 내부 운영 목표**(지원 티켓 응답·정비 처리 목표·가용성 — 위반=경보·개선 루프). 라벨·칩에서 항상 구분 표기 — 혼용 금지. 둘 다 **구성 가능한 설정 개체**여야 한다(임계·창·에스컬레이션 = no-code 편집·개정 스테이징 §3.9.0 — HO-01 문법). 참조: 지원 티켓=SLO(적용 ✓ 07-10), 현장·계약=SLA 유지, FW- 가용성=SLO.

## 5. 데이터·분석 원칙
- 모든 상태 전이는 이벤트로 남긴다: `(actor, timestamp, action, [linked objects], reason?)`.
- 흐름 완료(예: 급여 승인, 입사 확정)는 관련 개체 전체에 역참조를 남긴다 — 나중에 워크플로 스튜디오가 **개체 타입과 관계를 트리거/조건으로 사용**할 수 있도록.
- 파생 수치(충원율, 출근율, 주52h)는 항상 원본 개체로 drill 가능해야 한다.
- **통합 원칙 (integration)**: 모듈은 섬이 아니다. 새 기능은 반드시 (a) 관련 개체에 참조 토큰·링크 칩으로 연결되고, (b) 상태 전이 시 연결 개체에 역참조·이벤트를 남기고, (c) 상·하류(개체 체인)로 한 번의 클릭 이동 경로를 제공한다. "이 기능이 어떤 개체와 연결되는가"를 답하지 못하면 미완성이다.

## 6. 백로그 (깊이 우선)
1. 계약 → 포지션 → 정책 프리셋 편집기 (상속·오버라이드 시각화)
2. 공고 생성 시 포지션 조건 자동 채움 / 입사 확정 → 직원·타임테이블 자동 생성
3. 연장근로 신청 개체 (AT- ↔ AP- ↔ 급여 자동 반영 완결)
4. 날짜 개체 드래그 참조 (주간 캘린더 → 채팅/할 일/코멘트)
5. 탈락 사유 카탈로그 + 인재풀, 면접 일정 ↔ 내 일정 개체 연결
6. 관계 그래프 뷰 (객체 탐색 화면) — 개체 클릭 시 상·하류 체인 시각화
7. **데이터 통합(Data Integration)** — 결정적(no-AI) 인제스트 파이프라인: 소스(파일 11종+사진·영상·ZIP·임의+외부 API) → 파싱/OCR → 정제 → 분류·템플릿 → 매핑(신뢰도·출처) → 검증 → 온톨로지 적재. IngestJob(`DX-`)·Source·Template=개체, provenance/lineage. Foundry 벤치마크·Rust 백엔드. (HANDOFF §10)
8. **증거능력 보존 아카이빙 + 미디어/ZIP** — 문서·기록물이 임의 파일·사진·영상·ZIP 수용; 원본 불변(WORM)+무결성(해시·TSA)+연계보관성; 인콘솔 사진·영상 뷰어/플레이어·ZIP 트리 readonly; 아카이브 최적화는 파생본에만. (HANDOFF §11)
9. **인콘솔 오피스 편집기** — ONLYOFFICE/Euro-Office **헤비 포크**(감사·PBAC·컴플라이언스·보안·비밀보호를 편집기 내부에 통합) + 호스트 거버넌스(버전·롤백·승인 게시(패스키)·협업). (HANDOFF §12)
10. **메일 = mox 백엔드 + 자체 프런트** — 오픈소스 mox(Go·MIT, IMAP4·SMTP·SPF/DKIM/DMARC·MTA-STS·DANE·암호화 저장·webapi/webhooks) 메일 서버를 백엔드로, 커뮤니케이션 모듈에 자체 메일 UI(rail↔main). 감사·PBAC·컴플라이언스(법적 보존·저널링·e-discovery)·보안·DLP·비밀보호를 위해 mox 개조. (HANDOFF §14)
