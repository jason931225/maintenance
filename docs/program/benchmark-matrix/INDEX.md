# Source-Bounded Benchmark Matrix — Executive Summary

**14 modules × 7 primary vendors (+ secondary references), read through 5 independent lenses, adversarially reviewed (4 verifier passes, 23 web checks + code adjudication).**

Each `<module>.md` in this directory is a self-contained benchmark: an evidence-based read of our console (`web/src/console/**` + backend crates + `docs/program/console-program-ledger.md`), a capability matrix vs Palantir Foundry · Slack · Microsoft Teams · Asana · n8n · Rippling · SAP (with ServiceNow/Workday/Zendesk/Vanta/Drata/Windmill/Temporal/Cedar/Purview/Korean-groupware as module-specific references), per-vendor "how they'd build our module" paragraphs, a **Cross-cutting lens findings** section (the 5 lenses woven in + adjudicated contradictions), and a ranked steal-list. The 5 lens documents are in [`lenses/`](./lenses/).

In the structured capability-matrix comparison cells, vendor claims use `[V]` for source-verified statements or `[I]` for inference. Narrative synthesis outside those cells is not covered by a universal labeling promise and must be read with its cited sources and revision-bound repository evidence. "Ours" claims distinguish source presence from runtime or enterprise-readiness proof.

## Per-vendor patterns — what the cited product surfaces emphasize

- **Palantir Foundry — the ontology reference; our explicit north star.** Object/property (cell-level) policy; **Actions as the sanctioned mutation verb**; branching/proposal-as-PR schema governance; **markings that propagate down lineage**; search-around exploration + saveable Layouts; Functions-on-objects; Automate condition→effect where automation and humans fire the same verb. Nearly every Foundry "steal" is *"finish what we half-built."* Weakness for us: no 전자결재/근로기준법 model, US-gov origin.
- **SAP (S/4HANA · SuccessFactors · GRC · FSM · SBPA) — enterprise rigor, at the cost of weight.** The **document principle** (immutable balanced GL, ACDOCA), OB52 period control; the **SoD ruleset + risk-at-grant + UAR recertification** benchmark (GRC); release-strategy **amount/attribute-driven routing** + flexible-workflow parallel approval; effective-dated master data + **first-class Positions/org units**; AI dispatch scheduler + **Crowd Service** (FSM); decision-table approver determination (SBPA). Steal the *invariant discipline*, reject the config-priesthood.
- **Rippling — the attribute-cascade HRIS.** The **employee graph**; **attribute-bound permissions that auto-update on transfer** (Supergroups); **Workflow Automator** (any field is a trigger); rule-based PTO accrual; geofenced clock-in bound to payroll; 90-second pay-run. The cited surface emphasizes lifecycle-linked automation. Weakness: US-payroll-centric, Korea via EOR overlay (no 근로기준법/전자결재 native).
- **Slack — collaboration + triage velocity.** Activity-view triage (filter pills, clear-all); **swarm threads** (26% faster close, 64% backlog cut); **one-click inline decide**; Workflow Builder conditional branching (≤15 branches, ≤10 rules each, Business+/Grid); link unfurl; Discovery API + Legal Holds + DLP (Enterprise Grid). The cited surface emphasizes the human-coordination + notification-terminal loop; shallow on structured objects and governed config.
- **Microsoft Teams — the frontline + enterprise-compliance hub.** Activity feed; native **Approvals app + e-signature**; **Shifts** (geofenced clock-in + handover; note: approval marks unavailable, does *not* auto-remove); Purview retention/eDiscovery; 1,000+ Power Automate connectors. The cited surface emphasizes in-flow approvals + frontline mobile; data scatters across M365, 전자결재 needs third-party add-ons.
- **Asana — no-code work-graph ergonomics.** Customizable Home + My Tasks + Inbox; Forms→task intake; **Rules + Bundles** (reusable automation pushed to many scopes); reference custom fields; immutable audit log. The cited surface emphasizes time-to-first-chart + citizen-config; capped (100 fields/project), no as-of, no governance depth, explicitly *no* custom retention / legal hold.
- **n8n — the open automation reference.** Free-form node DAG; 400+ connectors + HTTP + code node; **inbound webhook trigger**; Error-Trigger workflow + per-node execution debugging; data pinning. The cited surface emphasizes raw engine breadth + debugging; untyped JSON items, no governance, prunes-not-preserves (the antithesis of custody).
- **Secondary references the lenses surfaced:** **ServiceNow** (configurable agent workspaces, GRC audit workspace, FSM Dispatcher Workspace) · **Workday** (BP-framework, effective-dating, Related Actions) · **Vanta/Drata** (continuous control-monitoring, hourly/daily) · **Windmill** (`permissioned_as`) · **Temporal** (durable event-sourced replay) · **Cedar** (policy templates, schema validation) · **Purview** (retention/disposition/eDiscovery) · **Zendesk** (macros, custom objects) · **Zapier** (linear simple-case path, 8,000+ apps) · **Concur/NetSuite** (1-click / bulk approve) · **Korean groupware 더존·다우오피스·한비로** (결재선, 함 IA, 전결/대결/후결).

**Bounded inference from the sampled product surfaces (observed 2026-07-18):** the cited connector-centric products emphasize outward extension (connectors, webhooks, apps, or code), while this candidate design emphasizes inward ontology-model extension. Inbound webhook, connector catalog, code node, and app platform remain deliberate gaps. The target differentiator is one shared governed Action shape for humans and automation; Cedar enforcement remains shadow/target work until each action is enrolled, proved, and promoted. This is a planning inference, not a market-wide fact.

## Candidate differentiators in the sampled comparison

1. **Governance is the substrate, not a bolt-on.** Current proof covers FORCE-RLS on evidenced tenant tables under the real `mnt_rt` role (not a BYPASSRLS superuser), legacy server-side authorization/middleware, and four-eyes `CHECK(approver_id <> requested_by)` at the DB. Cedar deny-by-omission to field/residual-SQL grain is the accepted target/shadow design and is not universal live enforcement until separately promoted.
2. **Audit-chain seal/verify/gap-detection plumbing (L20), currently DARK** — production sealing defaults OFF; the in-memory signer is dev/test-only and not a trust root; NULL-org events are excluded. Real tamper evidence requires an external signer plus an out-of-band anchor, and object-lock deployment proof remains pending. No superiority or universal-mutation claim follows from the current source.
3. **Config-as-governed-object.** Dashboards, SLO settings, console views (and, on the roadmap, policies + approval templates) are themselves versioned ontology instances with draft→approve→effective + rollback + as-of + four-eyes. Foundry config isn't a first-class versioned business object; the sampled support-vendor citations do not show its SLA/SLO config that way (a config change here is itself four-eyes-able).
4. **Object-level as-of / bitemporal on instances.** `get_as_of` + `history` + per-revision `hashVerified` reconstruct the *whole object* as-of T. Foundry's temporal story centers on time-series properties + edit-history transactions; a uniform object-level reconstruction is our native primitive.
5. **Automation uses the same declarative Action shape as humans.** Current workflow execution remains under legacy server guards and evidenced governance checks. Cedar inheritance and a defined `runs_as` principal are target requirements; no universal live Cedar gate or no-shadow-privilege claim is made yet.
6. **Honest-by-default UI (a design law, not a preference).** Unbacked sections are *omitted, never placeholdered*; axis-truncation is code-governed and forces a mandatory "축 절단" warn chip (anti-deceptive-chart law — the sampled citations do not establish an analogous mandatory disclosure chip); CVaR/tail-risk is source-implemented in the ops-dashboard code; the deliberate **NO-AI** stance keeps every source-evidenced surface deterministic and auditable.
7. **Korean-statutory primitives are a differentiated foundation.** Source evidence covers delegate-finalize/대행, post-finalization rejection + compensation, round-labelled 연차촉진 notices and receipts, the 4대보험/NTS/severance payroll kernel, per-shift 근로계약, PIPA/개인정보보호법, and Group→법인→branch→worksite scoping. Native 전결/대결/후결 plus statutory leave timing/sequence enforcement remain documented gaps rather than source-implemented capabilities.
8. **Real, RLS-tested finance period lock.** `period_lock.rs` (+ migration `0107`) blocks ledger writes into a closed period, tested as `mnt_rt` — source-present and runtime-role tested; the cited Rippling/Palantir surfaces are different models, not proof of superiority. (The governance & data-model lenses under-credit this; it stands.)
9. **A highly collapsed leave surface with a material open loop** — co-located request inputs, source-wired team-decision calls, and source-wired 촉진 calls, and ledger on one screen. Request creation, statutory timing/sequence enforcement, and closed-loop E2E must be fixed before propagation.

**Honest caveats that keep this list credible:** tenant seeding publishes **27 types (9 governed config + 3 C-chain + 15 projected domains)**, but breadth is not dynamic-layer depth. Projected types remain domain-owned for writes; only `registry.update_equipment` has a registered projected dispatch and unregistered targets fail closed. Several flagship strengths remain **intent, not yet law**, and most Korean IA moats (결재함/상신함 inbox/outbox/archive boxes, dispatcher board, main comms views, command palette results) remain unwired. Source presence is not deployment, browser, database, or enterprise-readiness evidence.
