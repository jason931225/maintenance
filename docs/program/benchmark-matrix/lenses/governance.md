## 0. OUR governance stack — the yardstick (evidence-based)

Read from the tree, not memory:

- **Permissions granularity — Cedar PBAC.** `console/policy/*` (PolicyGate, deny-by-omission), backend `cedar_pbac/{authoring,residual}.rs`. Object-policy (row) + property-policy (field); partial-eval → residual → SQL `WHERE` composed with RLS floor (`WHERE <RLS> AND <residual>`); forbid-wins; untranslatable term → collapse-to-DENY (fail-closed). Covert = not rendered. Real 3VL bug (absent JSONB attr under forbid) caught by `mnt_rt` suite → Cedar-faithful COALESCE. [ledger 2026-07-10 residual]
- **Tenant isolation — FORCE-RLS on every table**, `app.current_org` GUC, tested as real `mnt_rt` role (not BYPASSRLS superuser). `mnt-gate-rls-arming` + `tenant-isolation` CI gates. This is a *guardrail forbid that always wins* — architecturally stronger than app-layer scoping.
- **Approval / SoD / four-eyes.** Frontend `console/appr/*` = 전자결재 approval line (multi-node sequential, per-node state, comment-required, reason options, evidence-required attachment policy, object-link targets). SoD self-approval detected client-side (`lineHasSelfApproval`). Backend `crates/governance` (`0153_create_governance.sql`): `gov_approvals` `CHECK(approver_id <> requested_by)`, decider≠requester, `four_eyes_approved_conn` re-checkable in-tx (TOCTOU-safe), decide reads authoritative requester in-tx (spoofed-requester self-approval hole closed). §16 gate-chain `[Authority(Cedar) → SelfChecklist → FourEyes → EgressDlp]` fail-closed. Distinct-approver four-eyes on evidence hold apply/release.
- **Audit surfaces.** `console/audit/AuditFeed`; backend universal `with_audit` wrapper on every mutation, append-only. **Tamper-evident hash chain (L20, PR #204 on main)** — crate-owned canonicalization, gap-proof CoverageGap verify, corrupt-seal→verdict. Cedar decision log (`0159_create_cedar_decision_log.sql`) — every authz decision recorded. (Audit-chain seals live at `0100`/`0101`.)
- **Retention / holds / WORM.** `console/evidence/*` EvidenceCard (WORM/fixity/custody/legal-hold chips). Backend `crates/docs/rest`: SHA-256 fixity per WORM copy, custody chain, nullable RFC-3161 TSA (deferred), POST verify (re-HEAD each WORM copy + hash-normalize, audited), POST hold (apply protective / release = fail-closed distinct-approver four-eyes). Object-lock bucket. ISO 15489 / OAIS / FRE 902(14) framing.
- **Lifecycle governance.** `crates/governance` overrides: effective-dated versioned store, as-of query, draft-direct vs override(reason + four-eyes + before-audit), impact preflight, soft-archive gate (no hard delete). Config = governed ontology object (draft→approve→effective, rollback, as-of) not code constants.
- **Config-as-governed-object.** `support_slo_setting` + `console_view` seeded *through the engine* (staging v+1 / as-of); Cedar actions `console:configure` (internal-only) + `console:deploy` (민감정보 + clearance, fail-closed).

**Net posture:** our governance *primitives* (RLS-forbid tenant floor, Cedar deny-by-omission down to field, four-eyes CHECK at the DB, WORM+hash-chain audit) are **best-in-class-tier** — most vendors bolt governance ON; ours is the substrate. Our *gaps* are UX/coverage: no SoD **ruleset library** (SAP has 1000s of predefined risks), no **continuous control-monitoring** loop (Vanta/Drata), no **retention-schedule/disposition** engine (Purview), no **DSR/consent** workflow (OneTrust), no **access-review/recertification** campaign (SailPoint-class), and approval routing is per-workflow-definition not a reusable **BP-framework** (Workday).

### 1. overview

We render permitted subset but don't *summarize the governance system itself*.
- **Verdict: Behind on governance-posture summary; Ahead on enforcement.**
- **STEAL:** ① governance-posture strip (pending four-eyes count · active legal holds · Cedar denials 24h · overdue lifecycle reviews) → Vanta/Drata → trivial: it's an ontQuery widget over gov_approvals/holds/decision_log → **S**. ② "why can't I see X" affordance (deny-reason on request) → Cedar → **M** (needs safe non-leaking denial copy).

## TOP-10 CROSS-MODULE GOVERNANCE FINDINGS (ranked)

1. **SoD is enforced (self-approval CHECK) but there is no SoD RULESET.** SAP GRC's whole value is a *predefined library of conflicting-permission pairs* (create-vendor vs pay-vendor, initiate vs approve) + mitigation controls w/ test-plans. We block one person approving their own request; we don't detect "this person holds two roles that together enable fraud." **Build a governed SoD-ruleset object-type (conflict pairs over Cedar actions) + mitigation-control library.** Touches appr/finance/people/compliance. [V SAP] — **L, highest governance ROI.**

2. **Compliance-as-continuous-monitoring is entirely unbuilt, and we own every primitive.** Control→Test→Evidence + cron-diff (Vanta hourly/Drata daily) + cross-framework mapping is the one place a specialist vendor decisively leads. Our WORM evidence + automation-fires-actions + audit chain make it a modeling exercise, not new infra. **Model Control/Test/Evidence as object-types; a scheduled automation = a test-run producing a WORM evidence instance.** [V Vanta/Drata] — **L.**

3. **Comms governance (hold/retention/eDiscovery/DLP) is the biggest coverage gap vs Slack/Purview** — and the ledger already flags it as unbuilt. Our deliberate no-E2EE + append-only + WORM substrate makes messenger/mail *the easiest place* to add litigation hold (reuse the evidence four-eyes-hold model), per-channel retention (governed setting object), and eDiscovery export (custodian/date-scoped, watermarked per §13.1). [V Slack/Purview] — **M–L.**

4. **Foundry's marking-propagation-down-lineage is the one object-governance feature we lack.** We filter at query time (residual); Foundry marks a source cell and every derived dataset inherits the eligibility gate. For a conglomerate platform (민감정보 flowing 법인→branch→analytics) this is the difference between "filtered on read" and "sensitivity travels with the data." **Marking = property + forbid-policy; propagate along link-types.** [V Foundry] — **L.**

5. **Retention/disposition lifecycle is missing on top of WORM.** We hold indefinitely and can legal-hold; we lack Purview's *scheduled retention window → disposition-review (four-eyes) → governed dispose*. We have the soft-archive gate and four-eyes — this is wiring a retention-label setting object to the archive lifecycle. Regulatory-record lock tier (admins can't release) is a cheap add. [V Purview] — **M.**

6. **Workday's Business-Process framework should generalize our per-workflow approval into one reusable governed abstraction.** Today approval routing is defined per workflow-definition; Workday models *every* mutating event as `definition→steps(approval/checklist/integration/notify)→condition/routing→mandatory commit`. Our appr line + governance commit step is 80% there. Splitting initiate/approve/**view** into three Cedar actions per BP is a free governance win. [V Workday] — **L (leverage), S (the 3-action split).**

7. **Automation may run with author privileges — adopt Windmill's `permissioned_as ≠ created_by`.** If a workflow/automation executes with its author's live grants, a departed/demoted author's automations become a privilege-escalation and stale-authority hole. Give each definition its own effective-principal, Cedar-evaluated at run time. This is a security fix, not polish. [V Windmill] — **M.**

8. **No access-review / recertification loop.** Every mature governance stack (SailPoint, ServiceNow, SAP) periodically asks "does this principal still need this grant?" We grant via Cedar principal-attributes but never expire/recertify them. A campaign = automation over principal-attributes → four-eyes revoke, fully within our model. Pairs with the break-glass TTL-token backlog item. [I/V ServiceNow] — **M.**

9. **Governance posture is invisible at the top level.** Vanta/Drata/ServiceNow all lead with a posture dashboard; we hide-by-omission but never *summarize the governance system itself* (pending four-eyes, active holds, Cedar denials 24h, overdue reviews, failing controls). Cheapest high-visibility win: an ontQuery strip on overview over gov_approvals/holds/decision_log. [V Vanta] — **S.**

10. **Korean-statutory governance is a genuine, defensible moat — protect and extend it.** 전자결재 (결재선/전결/대결/후결/반려), 연차촉진 rounds, 수령확인/노무수령거부, per-shift 근로계약, PIPA/개인정보보호법 — global vendors (Workday/ServiceNow/SAP/Slack) *force-fit or omit* these. We already ship the shapes. The remaining local-governance builds (delegation 전결·대결, 합의 parallel co-sign, receipt-confirmation as governed acknowledgment, PIPA DSR w/ legal-hold check) are where we stay uncatchable — prioritize them over re-implementing generic approval features vendors already do. [V/I across appr/leave/compliance] — **M each, strategic.**
