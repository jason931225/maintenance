# Benchmark Matrix — Module: `policy` (PBAC authoring / simulation / decision logs / assignment receipts)

> **Benchmark evidence metadata**
> - Observation/revalidation date: 2026-07-19.
> - Sampled products/surfaces: Oyatie Cedar policy/simulation/log source; Palantir; SAP GRC; Rippling; Slack; Microsoft Entra/Teams; Asana; n8n.
> - Evidence modality: Fixed-target repository source plus live-checked public official documentation/product pages and explicitly labeled public secondary pages; hands-on product tenants, screenshot capture, deployment, activation, and production validation were not performed.
> - Scope/claim ceiling: Only the named pages, surfaces, and fixed-target source are in scope; no whole-product, current-production, provider-parity, universal-superiority, legal, tax, labor, deployment, activation, or production conclusion.
> - Legend: [V] = bounded external observation with a direct URL or same-document source-list entry; [E]/[code] = fixed-target repository observation; [I] = recommendation or inference. Every steal/adopt item is [I].

**Scope:** No-code Cedar policy authoring (P→R→A→E blocks), server-side simulation, an append-only decision feed, and the target deny-by-omission authorization architecture and role-assignment governance (roles = principal attributes, never policy). Accepted ADR-0021 explicitly leaves live authorization on the current path until each action is enrolled, shadowed, evidenced, and promoted. Compared against Palantir Foundry, SAP (GRC Access Control + S/4HANA), Rippling, Slack (Enterprise Grid), Microsoft Teams (Entra RBAC), Asana, n8n.

**Most-relevant vendors** (this module's real competitors): **Palantir** (granular row/marking policies + partial-eval→query), **SAP GRC** (SoD ruleset, risk analysis, access-request 전자결재), **Rippling** (attribute-driven permission profiles / Supergroups). The collaboration tools (Slack/Teams/Asana/n8n) are RBAC-only and mostly N/A on the deep PBAC dimensions — noted per row.

## OUR CONSOLE — evidence base (read from code, not claimed)

Labeled **[E]** = verified against this repository.

- **Authoring UI:** `web/src/console/policycanvas/` — `PolicyCanvasScreen.tsx` (38KB), `model.ts`, `types.ts`. Fixed **Principal → Resource → Action → Effect** block sequence on the shared `console/canvas/BlockCanvas`; effect ∈ `{permit, forbid}`; actions ∈ `{view, edit, read_field, console:configure, console:deploy}`; resource condition attrs whitelisted to `{resource_type, owner, branch, legal_hold}`; principal set-attrs `{roles, clearance_keys}` via `contains`. The authored payload IS the REST contract (`api/policyCedar.ts`, OpenAPI-generated) — the canvas "never invents fields the backend cannot store." **[E]** `types.ts:10-38`
- **No local evaluator — simulation is server-truth:** "The decision itself always comes from `POST /policy/simulate` — there is no local evaluator." The Phase-C wave-1 wire-up (ledger) explicitly *deleted* the local evaluator. **[E]** `model.ts:1-4`; ledger line 160
- **Backend Cedar substrate:** `backend/crates/platform/authz-rest/` exposes `/policy/simulate`, `/policy/authorize`, `/policy/authorize/bulk`, `/policy/decisions`, plus catalog/drafts/validate/submit/**review**. `cedar_pbac/authoring.rs` = blocks→normalized_row→Cedar text, strict-validate, **review FSM + four-eyes**, object-policy(row) + property-policy(field) eval, deny-by-omission, **forbid-wins**. These endpoints prove the authoring/evaluation substrate, not promotion of production routes to Cedar enforcement. **[E]** `authz-rest/src/lib.rs:61-76`; ADR-0021; ledger 152
- **Current versus target enforcement:** frontend gate `console/policy/authz.ts` mirrors `GET /me/authz`; `authority` is always `advisory_ui_only` — "Nothing here is a security boundary." The current backend authorization behavior and armed RLS remain the live security boundaries. Missing UI capability ⇒ **absent** (not a disabled ghost), with a fail-closed JWT-floor fallback on projection-endpoint outage. Cedar deny-by-omission is the target per-action path only after the ADR-0021 coexistence gates are satisfied. **[E]** `policy/authz.ts:12-29, 68-122, 185-198`; ADR-0021
- **Residual / list filtering:** `cedar_pbac/residual.rs` lowers our own condition grammar (`= ≠ ≥ ≤ ∈`, AND-within / OR-across-permits, forbid = AND-NOT-COALESCE) → **SQL WHERE**, composed with RLS floor (`WHERE <RLS> AND <residual>`); untranslatable term ⇒ collapse-to-deny. Cedar itself is a point-decision evaluator; row-filtering is our own lowering (arch D1). **[E]** ledger 99, 154
- **Decision log:** `0159_create_cedar_decision_log.sql` — append-only, FORCE-RLS, org-scoped ledger of `/policy/authorize[/bulk]` decisions (actor, subject_ref, action, resource, effect, determining_policies JSONB, reason). No UPDATE/DELETE. Read recent-first for the Integrity console via `/policy/decisions`. Retention prune noted but not built (`ponytail:`). **[E]** migration file
- **Role assignment = governance, not policy:** roles are Cedar principal *attributes* in the target model; the console API, `crates/governance` gate-chain, `CHECK(approver_id<>requested_by)` self-approval ban, and append-only `gov_approvals` provide governance substrate. This evidence does not establish universal Cedar enforcement or promotion of every assignment path. **[E]** `backend/crates/governance`; migration `0153`; ADR-0021
- **Tamper-evident receipts (partial):** the audit-chain crate and migrations `0100`/`0101` provide hash-chain seal/verify and gap-detection evidence. **[E]** The follow-up signer direction is a provider-neutral key-management/signing capability with the self-host reference implementation first; OCI Vault is only an OCI-context adapter example. **[I]** A dedicated signed role-assignment receipt object is not yet a first-class surface. **[E]** `backend/crates/platform/audit-chain`; migrations `0100`/`0101`
- **Korean fit (bounded):** four-eyes / SoD split, self-approval ban, and the governance approval chain are useful electronic-approval controls; `console/appr/` (ApprovalCompose) + `console/leave/` carry `receipt_status`; branch/법인-scoped `BranchScope` exists in the capability model. Native 전결/대결/후결 semantics and legally sufficient review artifacts remain gaps unless separately implemented and proved. **[E]** `policy/authz.ts:33-48`; ledger 172
- **Known gaps:** no SoD *conflict-ruleset* analysis engine (SAP-style "these two grants together = risk"); no access-request self-service catalog with risk-simulation-at-request; no periodic access **recertification** campaign; decision-log retention prune unbuilt; signed receipt object unbuilt.

## PER-VENDOR: "how they'd build OUR `policy` module"

**Palantir Foundry** — Would drop our fixed P→R→A→E form and instead make **every ontology object type carry its own restricted-view policy**, with markings as the first-class classification primitive (민감정보/PII/근로계약 = markings on rows). Authoring = pick "user's markings" vs a marking column; enforcement = template→query at read time (the same target shape as our residual→SQL WHERE). They'd lean harder on **mandatory control properties** so a field can't exist un-classified. Their weakness for us: no 전자결재 approval routing, no SoD ruleset — pure data-access governance. Our target architecture resembles Foundry-style governance on a Korean-operations ontology; current live-route parity is not claimed.

**SAP GRC** — Would build it as a **risk-first control tower**: a shared SoD ruleset (transaction↔auth-object conflict pairs), batch + real-time risk analysis firing the moment a grant is proposed, and an MSMP multi-stage access-request workflow with Firefighter emergency access and periodic UAR recertification. Authoring is config-heavy (consultant-grade), not a pretty canvas. Their version turns "simulate" from *point-decision preview* into *portfolio-risk preview* ("if you grant this, here are the 3 new SoD violations"). This is the selected governance reference in this seven-product sample and a source of features missing here (SoD ruleset, recert campaigns, request catalog).

**Rippling** — Would make the whole module **disappear into HR attributes**: no policy canvas at all — you define Supergroups (dynamic attribute lists) and attach permission profiles; as an employee is promoted or transfers 법인, access recomputes automatically. "Simulation" = preview Supergroup membership. Their philosophy: policies you *set and forget* because the org graph drives them. Strong on lifecycle-linked provisioning and mobile; weak on expressive per-row/field conditions and formal approval chains. Their version optimizes our biggest lifecycle gap (auto-grant/revoke on HR events). **[I]**

**Slack (Enterprise Grid)** — Would ship a clean **RBAC delegation console**: named custom roles = permission checkboxes, assign to individuals or IDP groups, a dedicated Roles-Admin, and an `admin.roles` API. No conditions, no simulation, no per-record grain. Their strength is *delegation ergonomics and role-management-as-a-product* (their engineering blog treats RBAC as core infra). Their version of our module would be beautiful and shallow — great assignment UX, zero PBAC depth.

**Microsoft Teams (Entra RBAC)** — Would push policy *out* of Teams into **Entra**: built-in roles, custom roles as permission lists, **administrative-unit scoping**, **PIM** for just-in-time time-bound elevation with approval, and **Access Reviews** for recert. Teams itself only consumes. Their version splits authoring (Entra) from the workload (Teams) — the opposite of our single-console philosophy, but PIM (time-bound grants) and Access Reviews are features worth stealing.

**Asana** — Would add org-level **custom admin roles** (Enterprise+ add-on) on top of object-level project permissions (Admin/Editor/Commenter/Viewer), governed by super-admins, with an **Audit Log API** to SIEM. No conditions, no simulation. Their version is RBAC + object-permission defaults enforced org-wide — closest to a "permission defaults governance" layer, useful as a model for our console-view/module-default policies.

**n8n** — Would model it as **project-scoped custom roles**: granular resource scopes (workflow.create/execute/share, credential.read…) toggled per role, assigned within a project boundary. Clean declarative scope lists, API-managed. Their version is the tidiest pure-RBAC scope grammar — a good reference for how we enumerate action scopes per ontology resource, but no PBAC conditions, simulation, or approval governance. **[I]**

## WHAT WE'D STEAL — ranked **[I]**

1. **SoD conflict ruleset + risk-at-grant simulation** → **SAP GRC** is the cited reference for this capability. Fit: extends our four-eyes/self-approval-ban control into declarative "grant A + grant B = risk" pairs, evaluated in the existing `/policy/simulate` path and surfaced at assignment time via the governance gate-chain. This is a high-priority current gap for Korean enterprise audit/규제. **Cost: L** (new ruleset store + conflict evaluator + report surface). **[I]**
2. **Attribute-driven auto-grant/revoke on HR lifecycle events (Supergroups)** → **Rippling**. Fit: ADR-0021 models roles as target Cedar principal *attributes*; wire HR promotion/transfer events, shadow-prove grant recomputation, and promote explicitly. Closes the "no auto-provisioning on promotion" gap. **Cost: M**. **[I]**
3. **Periodic access recertification campaigns (UAR)** → **SAP GRC** (Entra Access Reviews second). Fit: a lifecycle FSM over grants (due→reviewed→confirmed/revoked) on our existing governance + audit-chain, producing signed 수령확인-style receipts. **Cost: M**. **[I]**
4. **Self-service access-request catalog with MSMP-style multi-stage routing** → **SAP GRC**. Fit: our governance gate-chain already does the *control*; add a request object type + approver-routing on the workflow engine so it's 전자결재-native, not admin-only. **Cost: M/L**. **[I]**
5. **Just-in-time, time-bound grants (PIM)** → **Microsoft Entra**. Fit: add an `expires_at` + activation-approval to a grant; audited activation on the decision log. Strong for 민감정보/console:deploy clearance. **Cost: M**. **[I]**
6. **Signed role-assignment receipt object** → extends **our own** L20 chain + **Asana** Audit-Log-API-to-SIEM pattern. ## Cross-cutting lens findings (5 independent review lenses)

- **Task-flow:** money task = *author a policy, verify it before ship*. The P→R→A→E block canvas is correctly rich for branching but a **~5+ interaction** overkill for the 80% simple "deny X to role Y" rule. **Steal:** a linear "quick-rule" path (single P→R→A→E row) beside the canvas — the block model already serializes; render a 1-row form variant (added as Steal #10). Cost **S–M**. **[I]**
- **IA / layout:** advisory policy projection is **woven into many console surfaces** (actions/nav/stats gate on policy codes), but UI projection is not an enforcement boundary. **GAP:** no first-class **policy authoring workspace** (policy list → editor → test/simulation → decision trace). **Steal:** policy test/simulation playground + decision-trace panel → Cedar/OPA [M]; policy-list master-detail (policy → affected principals/actions) [M]; "why blocked" inline trace from a gated action [S]. **[I]**
- **Data-model:** the target model adds a no-code P→R→A→E canvas → normalized row → generated Cedar text, governed policy objects, and residual → SQL WHERE around Cedar's typed-entity model. The implemented substrate does not prove live-route enrollment. **Weaker:** Cedar's schema-validation of policies against entity types is a mature safety net; our residual fail-closes to DENY on any untranslatable term (safe but coarser). **Steal:** Cedar schema-based policy validation (catch type errors at author time) [M]; `commonTypes` reuse (shared predicate fragments) [S]. **[I]**
- **Governance:** the implemented change-governance substrate is four-eyes-gated, simulated, and versioned, but ADR-0021 still requires route-specific shadow and promotion evidence before Cedar becomes live. **Steal:** **Cedar policy templates** (parameterized reusable policy with typed `?principal`/`?resource` holes) — surfaced convergently by three lenses (added as Steal #9) [M]; ServiceNow tri-state field rules (Mandatory/Visible/Read-Only) as a governed-config layer distinct from Cedar authz [M]; CI schema-validation gate on every policy [S]. **[I]**
- **Automation / extensibility:** implemented extensibility is policy data-authoring; automatic ontology enrollment, HR lifecycle wiring, decision-event automation, and marketplace-style extension remain gaps. **Steal:** policy templates for discretionary grants ("share object X with user Y" without writing N policies) → Cedar [S–M]; decision-log stream (every authz decision as an audit/automation event — on-deny → notify) → OPA [M]; policy-as-automation-trigger (on policy-change → re-simulate affected surfaces) [M]. **[I]**

**Adjudicated additions:** the ranked steal-list gained two lens-convergent items — **Cedar policy templates** (governance + automation-ext + data-model all flagged it) and the **linear quick-rule path** (task-flow) — and the Palantir "mandatory control property" cell was corrected (it is a required-marking row/datasource-level gate, not a per-field classifier; object/property policies give field grain). **[I]**
