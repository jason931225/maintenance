# Benchmark Matrix — Module: `policy` (PBAC authoring / simulation / decision logs / assignment receipts)

**Scope:** No-code Cedar policy authoring (P→R→A→E blocks), live simulation, deny-by-omission enforcement, an append-only decision feed, and role-assignment governance (roles = principal attributes, never policy). Compared against Palantir Foundry, SAP (GRC Access Control + S/4HANA), Rippling, Slack (Enterprise Grid), Microsoft Teams (Entra RBAC), Asana, n8n.

**Most-relevant vendors** (this module's real competitors): **Palantir** (granular row/marking policies + partial-eval→query), **SAP GRC** (SoD ruleset, risk analysis, access-request 전자결재), **Rippling** (attribute-driven permission profiles / Supergroups). The collaboration tools (Slack/Teams/Asana/n8n) are RBAC-only and mostly N/A on the deep PBAC dimensions — noted per row.

## PER-VENDOR: "how they'd build OUR `policy` module"

**Palantir Foundry** — Would drop our fixed P→R→A→E form and instead make **every ontology object type carry its own restricted-view policy**, with markings as the first-class classification primitive (민감정보/PII/근로계약 = markings on rows). Authoring = pick "user's markings" vs a marking column; enforcement = template→query at read time (exactly our residual→SQL WHERE, which we already converge on). They'd lean harder on **mandatory control properties** so a field can't exist un-classified. Their weakness for us: no 전자결재 approval routing, no SoD ruleset — pure data-access governance. Their version is the most architecturally identical to ours; we're effectively building Foundry-governance on a Korean-ops ontology.

**SAP GRC** — Would build it as a **risk-first control tower**: a shared SoD ruleset (transaction↔auth-object conflict pairs), batch + real-time risk analysis firing the moment a grant is proposed, and an MSMP multi-stage access-request workflow with Firefighter emergency access and periodic UAR recertification. Authoring is config-heavy (consultant-grade), not a pretty canvas. Their version turns "simulate" from *point-decision preview* into *portfolio-risk preview* ("if you grant this, here are the 3 new SoD violations"). This is the single richest governance model of the seven and the clearest source of features we're missing (SoD ruleset, recert campaigns, request catalog).

**Rippling** — Would make the whole module **disappear into HR attributes**: no policy canvas at all — you define Supergroups (dynamic attribute lists) and attach permission profiles; as an employee is promoted or transfers 법인, access recomputes automatically. "Simulation" = preview Supergroup membership. Their philosophy: policies you *set and forget* because the org graph drives them. Strong on lifecycle-linked provisioning and mobile; weak on expressive per-row/field conditions and formal approval chains. Their version optimizes our biggest lifecycle gap (auto-grant/revoke on HR events).

**Slack (Enterprise Grid)** — Would ship a clean **RBAC delegation console**: named custom roles = permission checkboxes, assign to individuals or IDP groups, a dedicated Roles-Admin, and an `admin.roles` API. No conditions, no simulation, no per-record grain. Their strength is *delegation ergonomics and role-management-as-a-product* (their engineering blog treats RBAC as core infra). Their version of our module would be beautiful and shallow — great assignment UX, zero PBAC depth.

**Microsoft Teams (Entra RBAC)** — Would push policy *out* of Teams into **Entra**: built-in roles, custom roles as permission lists, **administrative-unit scoping**, **PIM** for just-in-time time-bound elevation with approval, and **Access Reviews** for recert. Teams itself only consumes. Their version splits authoring (Entra) from the workload (Teams) — the opposite of our single-console philosophy, but PIM (time-bound grants) and Access Reviews are features worth stealing.

**Asana** — Would add org-level **custom admin roles** (Enterprise+ add-on) on top of object-level project permissions (Admin/Editor/Commenter/Viewer), governed by super-admins, with an **Audit Log API** to SIEM. No conditions, no simulation. Their version is RBAC + object-permission defaults enforced org-wide — closest to a "permission defaults governance" layer, useful as a model for our console-view/module-default policies.

**n8n** — Would model it as **project-scoped custom roles**: granular resource scopes (workflow.create/execute/share, credential.read…) toggled per role, assigned within a project boundary. Clean declarative scope lists, API-managed. Their version is the tidiest pure-RBAC scope grammar — a good reference for how we enumerate action scopes per ontology resource, but no PBAC conditions, simulation, or approval governance.

## WHAT WE'D STEAL — ranked

1. **SoD conflict ruleset + risk-at-grant simulation** → **SAP GRC** does it best. Fit: extends our four-eyes/self-approval-ban control into declarative "grant A + grant B = risk" pairs, evaluated in the existing `/policy/simulate` path and surfaced at assignment time via the governance gate-chain. This is our biggest true gap and the highest-value differentiator for Korean enterprise audit/규제. **Cost: L** (new ruleset store + conflict evaluator + report surface).
2. **Attribute-driven auto-grant/revoke on HR lifecycle events (Supergroups)** → **Rippling**. Fit: roles are already Cedar principal *attributes*; wire HR promotion/transfer events (we own hr.rs + leave FSM) to recompute grants automatically. Closes the "no auto-provisioning on promotion" gap. **Cost: M** (event hooks from hr → role-attribute recompute; mechanism mostly exists).
3. **Periodic access recertification campaigns (UAR)** → **SAP GRC** (Entra Access Reviews second). Fit: a lifecycle FSM over grants (due→reviewed→confirmed/revoked) on our existing governance + audit-chain, producing signed 수령확인-style receipts. **Cost: M**.
4. **Self-service access-request catalog with MSMP-style multi-stage routing** → **SAP GRC**. Fit: our governance gate-chain already does the *control*; add a request object type + approver-routing on the workflow engine so it's 전자결재-native, not admin-only. **Cost: M/L**.
5. **Just-in-time, time-bound grants (PIM)** → **Microsoft Entra**. Fit: add an `expires_at` + activation-approval to a grant; audited activation on the decision log. Strong for 민감정보/console:deploy clearance. **Cost: M**.
6. **Signed role-assignment receipt object** → extends **our own** L20 chain + **Asana** Audit-Log-API-to-SIEM pattern. Fit: promote the append-only approval event to a first-class, OCI-Vault-signed receipt object (already the planned signer follow-up), exportable to SIEM. **Cost: S/M** (signer is already charted).
7. **Mandatory control properties (can't-exist-unclassified fields)** → **Palantir**. Fit: our ontology registry could require a classification/marking prop before a type publishes, so property-policy has something to bind. **Cost: S/M**.
8. **Role-management-as-a-product delegation UX** → **Slack**. Fit: polish assignment ergonomics (assign to IDP group, named delegated roles-admin) — pure UX borrow, no model change. **Cost: S**.
9. **Cedar policy templates** (parameterized reusable policy with typed `?principal`/`?resource` holes bound at attach — e.g. "manager-approves-own-team" instantiated per 법인) → **AWS Cedar** → *native fit*: a template = a governed policy doc with typed holes, instantiated per org unit without authoring N near-identical policies. Surfaced convergently by the governance + automation-ext + data-model lenses as a missing Cedar-native primitive. **Cost: M**.
10. **Linear "quick-rule" authoring path** (a single P→R→A→E row form beside the canvas for the 80% "deny X to role Y" case; canvas retained for branching) → **Cedar text ergonomics** → the block model already serializes; render a 1-row form variant. Task-flow lens measures the canvas at ~5+ interactions for the simple rule. **Cost: S/M**.

**Korean-context notes:** global RBAC tools (Slack/Teams/Asana/n8n) have no 전자결재 approval routing and no SoD — they mismatch 근로기준법-driven segregation and group-company (법인) scoping out of the box; SAP GRC fits the *control* model but is consultant-heavy and not ontology-native. ## Cross-cutting lens findings (5 independent review lenses)

- **Task-flow:** money task = *author a policy, verify it before ship*. The P→R→A→E block canvas is correctly rich for branching but a **~5+ interaction** overkill for the 80% simple "deny X to role Y" rule. **Steal:** a linear "quick-rule" path (single P→R→A→E row) beside the canvas — the block model already serializes; render a 1-row form variant (added as Steal #10). Cost **S–M**.
- **IA / layout:** policy gating is **woven into every surface** (actions/nav/stats gate on policy codes) — ambient enforcement more mature than any vendor's bolt-on. **GAP:** no first-class **policy authoring workspace** (policy list → editor → test/simulation → decision trace). **Steal:** policy test/simulation playground + decision-trace panel → Cedar/OPA [M]; policy-list master-detail (policy → affected principals/actions) [M]; "why blocked" inline trace from a gated action [S].
- **Data-model:** **stronger than raw Cedar/OPA** — we *use* Cedar's typed-entity model and add a no-code P→R→A→E canvas → normalized_row → generated Cedar text, governed policy objects (draft→approve staging), and partial-eval → residual → SQL WHERE (Cedar's `is_authorized_partial` is experimental — we lowered our own grammar). **Weaker:** Cedar's schema-validation of policies against entity types is a mature safety net; our residual fail-closes to DENY on any untranslatable term (safe but coarser). **Steal:** Cedar schema-based policy validation (catch type errors at author time) [M]; `commonTypes` reuse (shared predicate fragments) [S].
- **Governance:** **Ahead/Par** — policy-change is four-eyes-gated + simulated + versioned (stronger than raw Cedar, which has no built-in change-governance). **Steal:** **Cedar policy templates** (parameterized reusable policy with typed `?principal`/`?resource` holes) — surfaced convergently by three lenses (added as Steal #9) [M]; ServiceNow tri-state field rules (Mandatory/Visible/Read-Only) as a governed-config layer distinct from Cedar authz [M]; CI schema-validation gate on every policy [S].
- **Automation / extensibility:** extensibility here is intentionally data-authoring (no app marketplace applies), which we've nailed. **Steal:** policy templates for discretionary grants ("share object X with user Y" without writing N policies) → Cedar [S–M]; decision-log stream (every authz decision as an audit/automation event — on-deny → notify) → OPA [M]; policy-as-automation-trigger (on policy-change → re-simulate affected surfaces) [M].
